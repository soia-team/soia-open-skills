# soia-safe-audit-fix-codebase

对一个完整代码库执行“先审计、后确认、再修复”的安全闭环。第一次运行只能读取和报告；修复阶段必须由客户在看到具体发现、文件、行为变化、测试与回滚方案后明确授权。

## 客户可读说明

### 这个技能可以做什么

| 客户想要 | 技能会做 | 客户能看到 |
|---|---|---|
| 给一个代码路径做整体安全检查 | 读取仓库规则、架构、依赖、入口、权限边界和代码实现 | 项目安全概览、攻击面、Findings、证据与覆盖缺口 |
| 判断某漏洞是否影响项目 | 对照依赖、版本、配置、部署方式和调用路径 | 命中/不命中/未知结论及证据链 |
| 修复审计发现 | 先给修复批次预览，等待确认后逐条修复 | 精确文件范围、diff、测试与逐项状态 |
| 验证修复是否真实有效 | 复现、直接测试、相关回归、重扫和独立复核 | 实际运行的命令、结果、未覆盖项和残余风险 |

### 客户如何使用

至少提供代码库路径。可选提供：目标分支/提交、生产部署方式、技术栈、威胁重点、允许使用的本地扫描器、测试命令、报告保存位置和可接受的修复窗口。

```text
使用 soia-safe-audit-fix-codebase 审计 <repo-path>。
第一阶段只读，给我整体报告和分批修复方案；我确认具体批次后再改代码和测试。
```

即使客户第一句话写“审计并修复”，也必须完成第一阶段并停在确认门，不能在同一阶段直接修改代码。

### 依赖与安装

安装本技能及其同仓依赖：

```bash
npx skills add soia-team/soia-open-skills -g -a '*' -s soia-safe-audit-fix-codebase -y
```

| 依赖 | 类型 | 安装 / 配置 | 缺失时怎么处理 |
|---|---|---|---|
| `soia-dev-fix-loop` | 强依赖 | 随本仓库安装 | 审计可继续；修复阶段停止，不能手写简化版闭环 |
| `soia-dev-coding-protocol` | 强依赖 | 随本仓库安装 | 修复阶段停止，避免无验证或扩大范围 |
| `soia-safe-track-vulnerability-intel` | 可选增强 | 随本仓库安装 | 仍可做代码审计，但依赖情报覆盖标为降级 |
| 项目原生构建/测试工具 | 强依赖于验证 | 按仓库文档安装 | 只能报告未验证修复，不得称为通过 |
| SAST/SCA/SBOM 工具 | 可选增强 | 使用客户已批准的本地工具 | 用原生依赖树、锁文件和人工审计降级，列出缺口 |

本技能默认不需要私有配置。工具偏好可放：

```text
~/.config/soia-skills/soia-open-skills/soia-safe/soia-safe-audit-fix-codebase/config.yml
SOIA_SEC_AUDIT_CODEBASE_CONFIG_FILE=<custom-config-path>
```

不得在配置、报告或日志中保存 Token、私钥、Cookie、数据库连接串、源码正文或生产数据。

### 日志与完成回执

审计阶段报告读取范围、排除范围、语言/构建系统、依赖和入口数量、实际运行的扫描器、Finding 数量、误报复核、覆盖缺口及代码树是否发生变化。

修复阶段按 Finding 报告 `fix/reject/defer`、修改文件、测试、重扫、独立复核和残余风险。提交、push、PR、部署和发布不包含在修复授权中，需另行明确授权。

## 不可跨越的确认门

阶段 A 是只读审计，阶段 B 才能修改。完整规则见 [确认门契约](references/confirmation-gate.md)。

阶段 A 结束时必须给出：

1. 审计基线：仓库路径的泛化标识、分支/commit、工作树状态；
2. Finding 表：稳定编号、严重度、`file:line`、证据、影响和置信度；
3. 修复批次：包含哪些 Finding、拟修改哪些文件、行为变化、测试、回滚和风险；
4. 明确提问：要求客户确认一个或多个批次/编号。

有效确认示例：

```text
确认修复批次 B1，包含 F-001、F-003；允许修改列出的 4 个文件并运行报告中的测试。
```

如果代码、分支、commit、工作树或修复范围在确认后发生实质变化，原确认失效；重新生成受影响批次并再次确认。

## 阶段 A：只读项目安全审计

### 1. 冻结范围和现场

解析目标路径并确认它是预期仓库。读取根目录和相关子目录的 `AGENTS.md`、贡献指南、安全策略、构建与测试说明。记录 Git 分支、commit 和已有未提交变更；不得清理、覆盖、暂存或修改客户改动。

定义审计包含与不包含内容。例如：源代码、依赖、容器/IaC、CI/CD、配置样例、鉴权边界；排除第三方 vendored 代码、生成目录、二进制和真实秘密文件。遇到符号链接、子模块和多仓目录时显式列边界。

### 2. 建立项目清单

运行只读盘点：

```bash
python3 scripts/inventory_codebase.py <repo-path>
```

脚本只输出语言、清单文件、构建入口、容器/IaC 和测试线索，不读取或上传秘密内容。盘点结果只是导航，不是安全结论。

### 3. 建立攻击面和数据流

识别外部入口、信任边界、身份与权限、数据存储、外部调用、反序列化、文件处理、命令执行、后台任务、管理端和部署边界。优先审查高风险数据流，不对所有文件平均用力。

按技术栈读取 [安全审计清单](references/audit-checklist.md)，只应用相关条目。框架或库的行为需要查官方文档、源码或锁定版本，不能靠记忆断言。

### 4. 审计依赖与已知漏洞

检查直接依赖、传递依赖、锁文件、最终 JAR/WAR/二进制、容器镜像和 SBOM。若 `soia-safe-track-vulnerability-intel` 可用，传入精确包名、版本、PURL/CPE 和部署条件做关联；否则使用项目原生依赖树和已批准的本地 SCA，并在报告中标记情报覆盖降级。

依赖存在不等于可利用。将版本命中、调用可达、配置、部署方式和外部暴露分别列证据。

### 5. 形成并对抗式复核 Findings

每条 Finding 必须包含：

- 稳定编号、严重度和置信度；
- `file:line` 或依赖/制品位置；
- 可观察的危险行为与攻击者所需前提；
- 影响范围和受影响资产；
- 最小安全复现或静态证据；
- 修复方向、测试建议和来源。

在报告前尝试推翻每条发现：检查输入是否真实可控、危险路径是否可达、框架是否已有编码/校验、部署是否满足条件。无法证实时降级置信度或列为“待确认”，不得用扫描器标题冒充事实。

### 6. 输出报告和修复预览，然后停止

按 [项目安全报告契约](references/report-contract.md) 输出管理摘要、架构与攻击面、Findings、依赖风险、覆盖缺口和修复批次。

确认以下事实后结束阶段 A：

- 没有修改、格式化、生成或删除代码；
- 没有运行可能改变依赖锁、数据库、远端或生产状态的命令；
- 客户已看到修复文件、行为变化、测试和回滚预览；
- 已明确等待确认，尚未进入阶段 B。

## 阶段 B：确认后的修复与验证

### 1. 重新核对确认和基线

只处理客户确认的 Finding/批次。重新读取分支、commit、工作树和目标文件；发现漂移时停止并刷新预览。已有无关改动必须保留，不得纳入补丁。

### 2. 建立安全复现

优先使用单元测试、集成 fixture、静态断言或本地 canary。不得在生产环境执行 PoC、端口扫描、爆破、真实命令执行、数据外传或破坏性验证。无法安全复现时写明原因和替代证据。

### 3. 调用修复闭环

把确认的 Findings 交给 `soia-dev-fix-loop`，并遵循 `soia-dev-coding-protocol`：逐条选择 `fix/reject/defer`，实施最小修复，搜索同类模式，禁止用 TODO、吞错、关闭规则、降低断言或删除测试伪装修复。

修复范围扩大到新的文件、公开 API、数据库、认证模型或部署架构时，先更新批次并重新确认。

### 4. 分层验证

每个 `fix` 至少具备：

1. 直接验证：原危险行为不再成立；
2. 正常路径：合法输入和业务行为仍成立；
3. 相关回归：调用方、模块或服务测试；
4. 重扫：原规则/SCA/SAST 不再命中，且不是被忽略；
5. 独立复核：检查 diff、反例输入、下游调用或另一审查视角。

测试工具无法运行、环境不完整或全量回归成本未获授权时，结论只能是“部分验证”或“blocked”，不能称为测试通过。

### 5. 交付修复回执

按 Finding 列出决策、代码变化、直接测试、回归、重扫、独立复核和残余风险。检查工作树，确认未混入未授权文件。

提交、push、建 PR、合并、部署或发布都需要独立授权；“确认修复”只授权本地修改与预览中列出的测试。

## 私密信息与中间数据

- 审计默认全程本地运行，不把源码、制品、SBOM、路径、依赖清单或报告上传给第三方 AI/扫描服务。
- 需要云端扫描器时必须先说明发送字段、目标服务和保留策略，并取得客户授权。
- 报告默认留在会话；保存时使用客户选择的目录。中间数据使用 OS 临时目录并在成功或失败后清理。
- 日志和报告对 Token、私钥、密码、连接串、Cookie、内部主机名、用户数据和绝对路径脱敏；证据只保留定位所需最小片段。

```bash
python3 scripts/inventory_codebase.py --self-test
python3 scripts/inventory_codebase.py <repo-path> --pretty
```

真实前向测试使用一个临时 fixture 仓库：放入源码、manifest、测试、CI、容器文件、忽略目录和一个只按名称识别的秘密候选；运行盘点后核对目标树未变化、忽略目录未进入统计、秘密内容和绝对路径未出现在 JSON。不得为了测试本技能而对真实生产仓库进入阶段 B。

技能自身还必须验证：阶段 A 前后目标仓库 diff/状态一致；阶段 B 只在有效确认后产生 diff；每条已修复 Finding 都有直接测试和至少一条独立复核证据。
