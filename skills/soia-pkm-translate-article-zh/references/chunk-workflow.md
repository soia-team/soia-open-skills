# 分块翻译工作流（normal / refined 长文场景）

本文档是 [SKILL.md](../SKILL.md) 分块相关步骤的技术细节展开。只在源文档触发分块阈值（默认 4000 词，见 config 的 `chunk_threshold`）时才需要读这份文档；短文直接在单次上下文内翻译，跳过本节。

## 两层架构

- **机械层**（`scripts/chunk_markdown.py`）：只负责按标题层级和段落边界把 Markdown 切成词数可控的块，纯 Python stdlib，不做任何翻译判断。
- **认知层**（调用本技能的 Agent）：读取块清单，负责术语抽取、翻译、术语一致性检查、润色——这些都在 Agent 的 LLM 上下文里完成，脚本不参与。

不要试图让脚本"顺便"翻译或调用任何翻译 API——分块和翻译永远是两个独立步骤。

## 步骤

### 1. 判断是否需要分块

```bash
python3 scripts/chunk_markdown.py --file <path.md> --json
```

输出的 `triggered` 字段为 `true` 时才需要走本文档的多块流程；为 `false` 时 `chunks` 只有一项，直接整篇翻译即可，不需要再往下走。

### 2. 抽取术语、合并术语表

在真正翻译任何一块之前：

1. 通读全文（`--json` 输出里每个 chunk 自带 `text` 字段，或直接读源文件），列出专有名词、技术术语、反复出现的短语。
2. 与 config 里的 `glossary`（原文 → 固定译法）合并，形成本次翻译的**会话术语表**——原文中出现但 config 未覆盖的术语，先给出建议译法，供后续所有块统一复用。
3. 会话术语表是本次翻译期间贯穿所有块的唯一术语依据，不要每块单独重新决定译法。

### 3. 组织每块的共享上下文

给每个块的翻译都附上同一份"共享上下文"，至少包含：

- 目标语言 / 受众 / 风格（来自 config 或本次确认结果）
- 会话术语表（步骤 2 的结果）
- 全文体裁与语气判断（步骤 1 之前的"先分析"结论）
- 当前块的位置信息：第 N 块 / 共 M 块，以及前后块大致讲了什么（帮助块首尾衔接自然，不要孤立翻译）

### 4. 逐块翻译

- 有并行 subagent 能力时，可以每块派发一个子任务并行翻译，都携带步骤 3 的共享上下文；没有并行能力时按顺序在同一上下文里逐块翻译。
- 每块翻译只处理自己范围内的文本，不要重复输出别的块的内容。

### 5. 合并

按原顺序拼接所有块的译文；如果原文档有 YAML frontmatter，按 [SKILL.md 的 frontmatter 规则](../SKILL.md) 单独处理，不经过分块。

### 6. 中间产物不进 vault

`chunk_markdown.py --output-dir <dir>` 会把 `chunk-NN.md` 和 `frontmatter.md` 写到 `<dir>/chunks/`，方便需要物理文件的并行翻译场景（例如派发给多个子任务分别读取自己的分块文件）读取。

- `<dir>` 应指向临时/暂存目录（例如系统临时目录），不要指向 Obsidian vault 内的可见位置。
- 翻译完成、合并成最终译文后，中间的 `chunks/` 目录应视为一次性工作文件，不属于最终交付物，最终只保留 `<原文件名>-中文版.md` 这一个文件在 vault 里。
- 如果没有并行需求，完全可以不传 `--output-dir`，只用 `--json` 输出的块清单在对话上下文里逐块翻译，不产生任何磁盘文件。

## 常见问题

| 现象 | 处理 |
|---|---|
| 单个代码块本身超过单块上限 | 脚本保证代码块不会被从中间切开，会整体保留在所在的块里，属于正常现象，不是 bug |
| 分块导致标题和正文分离到不同块 | 脚本按小节（标题开始到下一个标题前）分组，只有小节本身超出上限才会在小节内部再按段落/行拆分；跨块衔接生硬时，翻译时利用步骤 3 的"前后块摘要"补上下文 |
| 总词数刚好在阈值附近 | 阈值只决定"要不要走分块流程"，不代表词数一超过 4000 就一定拆成多块——只要全文能塞进一个不超过 `chunk_max_words`（默认 5000）的块，`chunk_count` 仍然是 1 |
