# soia-dev-prompt-clarity

> 属 SOIA 开发者体验域（`soia-dev-*`）的沟通辅助技能：面向"人向 AI 表达请求"这一步。写提示词是与 AI 协作的基本功——本技能覆盖从零起草、诊断优化、防误伤改写三种模式，不产生代码、不写文件，只输出可直接使用的提示词文本和说明。

## 第一性行为准则：澄清优先于产出（贯穿三模式）

**看不懂就问，绝不硬编。**无论哪种模式，动笔之前先确认三件事是否清楚：

1. **意图**：用户到底想让 AI 完成什么？做完之后拿结果去干什么？成功标准是什么？
2. **目标受众 AI**：这个提示词要发给什么 AI、什么场景——通用对话助手、编码 agent、图像生成、还是某个特定工具？不同目标的写法不同。
3. **期望输出形态**：用户想要什么样的结果——一段文字、一张表、一个文件、一份代码、还是一场多轮对话的开场？

任何一项不明确，**不要猜着写**——返回一份具体的澄清问题清单：缺什么问什么、一次问全，不要挤牙膏式地一轮问一个。等用户补充后再产出。

- 判断标准：如果两个合理的人读同一句输入，会写出方向不同的提示词，就说明信息不够，必须先问。
- 唯一例外：缺口不涉及意图 / 受众 / 输出形态三项主干、只是次要细节（如具体名称、数量）时，可以先产出并用显式 `<占位符>` 标注，同时列出待确认项——但必须告诉用户哪些是占位、为什么。

## 三种工作模式总览

| 模式 | 什么时候用 | 输入 | 输出 |
|---|---|---|---|
| **A · 从零写提示词**（基本功） | 用户说「帮我写个提示词做 X」，或给一段模糊需求 | 需求描述（可能不完整） | 按七要素框架组织、可直接复制使用的结构化提示词 |
| **B · 优化已有提示词** | 用户已有一版提示词，效果不好或想返工 | 现有提示词全文（最好附"哪里不满意"） | 六维诊断表 + 改写版 + 逐条改动说明 |
| **C · 防误伤改写**（专项） | 一句正当请求被安全分类器误判、拦截或导致降级 | 被误报的原句 | 四诱因诊断 + 上下文补全后的改写句 + 改动说明；命中红线则只给问题说明 |

模式判定由 Agent 根据输入自动进行；拿不准时先问用户"你是想新写、改旧、还是这句被误报了"。三种模式共用同一条第一性行为准则和同一份回执格式。

> **与 soia-dev-task-prompt 的分界**：用户说「写提示词」时，默认命中的是**本技能**——包括给调研、探索、派发类任务从零写 prompt（走模式 A）。只有一种情况例外：在 SOIA 提案治理流程里做 task 拆分、需要带架构契约不变量与 Acceptance 的提案 task prompt，那才是 `soia-dev-task-prompt` 的职责。

## 交付方式：仅产出（默认）or 产出并执行

**默认仅产出**——写好的提示词完整展示给客户，由客户自行决定拿去哪里用。

仅当客户在**当前消息**明确说「写完就执行」「写好直接跑」「发给 codex / gemini / 子代理执行」时，进入**产出并执行**：

1. **先展示后派发**：照常完成澄清与产出，把提示词全文展示给客户后才派发——执行不黑箱。
2. **选执行器**：客户指定的 AI CLI（codex / gemini / kimi 等，派发、workdir 隔离与后台监控遵循 `soia-dev-coding-agent` 的执行器规范）／运行时子代理／当前会话自己执行。客户没指定且有多个可选时，问一次，不自作主张。
3. **执行闸门**：提示词内容含写文件、删除、发布、推送等风险动作时，派发前报一遍「执行器 + workdir + 风险动作清单」再等确认——**「写完就执行」授权的是派发这个动作，不自动等于授权提示词里的全部风险动作**。纯只读/纯产出类提示词可直接派。
4. **回传结果**：执行结束后给结果摘要、产物位置、执行器原始回执要点；失败如实报失败信号，不粉饰、不由本技能代为兜底改写结果。

确认语判定：「写吧」「先写着」「写一份」＝仅产出；「写完就执行」「直接跑」「发给 codex」＝产出并执行。

## 客户可读说明

### 这个技能可以做什么

覆盖提示词工作的三种模式：从零写出结构化提示词（模式 A）、诊断并优化已有提示词（模式 B）、把被安全分类器误判的正当请求改写成上下文完整无歧义的表达（模式 C）。**默认只产出提示词文本**（不调外部 API、不写文件）；客户明确说「写完就执行」时，产出后可派发给指定 AI 执行（见「交付方式」一节），执行阶段的动作以提示词内容与执行器规范为准。

| 客户想要 | 技能会做 | 客户能看到 |
|---|---|---|
| 从零得到一个提示词（模式 A） | 先做澄清检查，再按七要素框架（目标/角色/上下文/约束/格式/样例/边界）组织 | 可直接复制使用的提示词全文 + 每个要素为什么这么写的说明 |
| 优化已有提示词（模式 B） | 按六个通用维度诊断（目标/上下文/格式/约束/矛盾/篇幅），只改有问题的部分 | 诊断结果 + 改写版全文 + 逐条"改了什么、为什么" |
| 防误伤改写（模式 C） | 按四类诱因诊断缺失上下文，先过红线检查，只用客户确认属实的信息补全 | 诊断 + 改写句 + 改动说明；命中红线时只有问题说明，没有改写版 |
| 输入信息不足 | 不猜、不硬编，按缺失项生成澄清问题清单，一次问全 | 一份具体的问题清单，答完即可产出 |
| 写完直接执行（需明说） | 先完整展示提示词，再按客户指定派发给 codex/gemini/子代理等执行器；含风险动作时先过执行闸门 | 提示词全文 + 执行器与 workdir 说明 + 执行结果摘要与产物位置 |

### 客户如何使用

1. 把需求、现有提示词或被误报的原句交给本技能；顺手说明目标 AI 和期望产出形态（知道多少说多少，缺的会被问到）。
2. Agent 判定命中哪种模式（A 新写 / B 优化 / C 防误伤）；判定不了时先问，不硬猜。
3. **澄清优先**：意图、目标受众 AI、期望输出形态任何一项不明，客户会先收到一份一次问全的澄清清单，而不是一版猜出来的产出。
4. 默认交付只输出文本（提示词 / 诊断 / 改写句 / 说明），不写文件、不发网络请求、不操作账号。只有客户当前消息明确要求「写完就执行」时才进入派发执行，且执行前有独立的风险闸门（见「交付方式」）。
5. 最终回复必须包含：模式判定、澄清记录（问了什么或"输入已足够"）、产出全文、逐条说明；模式 C 命中红线时，回复只包含问题说明，不包含改写版。

## 模式 A · 从零写提示词（基本功）

用户给一句"帮我写个提示词做 X"或一段模糊需求，本模式产出一版结构完整、可直接复制使用的提示词。

### 提示词要素框架（七要素）

| # | 要素 | 回答的问题 | 是否必填 |
|---|---|---|---|
| 1 | **任务目标** | 让 AI 做什么、做到什么程度算成功 | 必填 |
| 2 | **角色设定** | AI 以什么身份/专业视角来做（编辑、评审、教练……） | 任务需要专业视角时填 |
| 3 | **背景上下文** | AI 需要知道哪些前情才能做对：对象是什么、给谁看、之前发生了什么 | 必填 |
| 4 | **约束条件** | 必须遵守什么：语言、篇幅、风格、不许做什么 | 必填 |
| 5 | **输出格式** | 结果长什么样：段落/表格/列表/代码/JSON，字段有哪些 | 必填 |
| 6 | **样例（few-shot）** | 给 1–3 个输入输出示范，让 AI 对齐口径 | 可选；格式敏感或口径微妙时强烈建议 |
| 7 | **边界与拒绝条件** | 遇到什么情况应该停下来问、标注不确定、或直接拒绝 | 建议填；有事实准确性要求时必填 |

要素不是模板填空：不需要的要素直接省略，不写空话凑格式。**判断标准是"删掉这一条，AI 的行为会不会变"——不会变的就删。**复杂度分级同样适用：简单任务通常 ≤3 个要素就够，中等任务 4–5 个，复杂任务才需要拉满 6–7 个，不为凑格式给简单任务硬填满七要素。

### 流程

1. **收集需求**：读用户的原始描述，对照第一性行为准则做澄清检查（意图 / 受众 AI / 输出形态）。
2. **澄清（如需）**：缺主干信息就出澄清清单，一次问全；只缺次要细节则用 `<占位符>` 先行产出。
3. **按七要素组织**：把用户给的信息归入对应要素，缺省的可选要素跳过。
4. **输出**：一版可直接复制的提示词（用代码块包住方便复制）+ 一段简短的要素构成说明（哪个要素对应用户哪句话、哪些是占位待补）。

### 示例

输入（模糊需求）：「帮我写个提示词，让 AI 帮我润色文章。」

澄清后（用户补充：发给通用对话 AI；文章是中文科普稿，面向普通读者；要保留口语感；想要改后全文加修改清单），产出：

```text
你是一名中文科普编辑。请润色我接下来提供的科普文章草稿。

背景：文章面向没有专业背景的普通读者，风格口语化，目标是易读且不失准确。

要求：
1. 保留原文的口语感和个人风格，不改成书面腔；
2. 只修正病句、冗余和逻辑断裂，不改动观点和事实；
3. 专业术语第一次出现时，补一句通俗解释。

输出格式：先给润色后的全文，再附一张修改清单表（位置 / 原文 / 改后 / 理由）。

边界：拿不准是否属于事实错误的地方，单独标注出来问我，不要直接改。
```

要素构成：任务目标=润色科普草稿；角色=科普编辑；上下文=读者与风格定位；约束=要求 1–3；输出格式=全文+清单表；边界=事实存疑先问。样例（few-shot）本例省略——格式已足够明确。

## 模式 B · 优化已有提示词

用户已有一版提示词，觉得效果不好、想返工，或单纯想让它更可靠。本模式先诊断、后改写，只改有问题的部分，不为改而改。

### 通用诊断维度（六项）

| # | 维度 | 检查问题 | 常见症状 |
|---|---|---|---|
| 1 | **目标是否明确** | AI 读完知道"做什么、做到什么程度"吗？ | 只有话题没有动作（"关于 X 帮我看看"）；一句话塞多个目标 |
| 2 | **上下文是否充分** | AI 拿到了做对这件事所需的前情吗？ | 用户心里清楚但没写出来的背景（给谁看、什么场合、之前试过什么） |
| 3 | **输出格式是否指定** | 结果的形态、结构、长度有说法吗？ | "帮我总结一下"——不知道要三行还是三页、要点还是成文 |
| 4 | **约束是否可执行** | 每条约束是 AI 能落实、能自查的吗？ | "专业一点""高级感"这类无法验证的形容词；该量化的没量化 |
| 5 | **指令是否互相矛盾** | 有没有两条指令打架？ | "尽量详细"+"控制在 100 字内"；"严格按模板"+"发挥创意" |
| 6 | **篇幅是否臃肿（长 ≠ 好）** | 每句话都改变 AI 行为吗？ | 重复叮嘱同一件事、堆砌礼貌用语和装饰性排比、粘贴大段无关背景 |

诊断只报有问题的维度，通过的维度一句话带过；六项全过就如实告知"不需要改"，不制造工作量。

### 流程

1. **通读原提示词**：先弄清用户想让它做什么、现在的效果差在哪（用户没说就问）。
2. **六维诊断**：逐项检查，输出诊断表——每个问题维度给出具体症状定位（引用原文哪一句）。
3. **澄清（如需）**：诊断发现的缺口需要用户补充事实（如缺背景、目标不明）时，按第一性行为准则先问再改。
4. **改写**：产出完整改写版（可直接复制），保留原文没问题的部分，不做无关的风格化重写。
5. **逐条改动说明**：每处改动对应到诊断维度——改了哪句、补了什么、删了什么、为什么。

删减和增补同样重要：模式 B 的产出经常比原文**短**。删掉不改变行为的句子、合并重复指令，本身就是优化。

### 与模式 C 的衔接

诊断中若发现原提示词属于防误伤场景——含敏感词（cookie / token / 抓取 / injection 等）且所有权、授权、用途上下文缺失——转入模式 C 流程处理该部分：先过红线检查，再按四原则补全。模式 C 的红线声明对模式 B 同样有效。

## 模式 C · 防误伤改写（专项场景）

一句**正当请求**被 AI 平台的安全分类器误判——拦截、追问或导致模型降级——本模式把它改写成上下文完整、意图无歧义的表达。

背景：AI 平台的安全分类器目前刻意设置得比较宽泛，官方原话是"safeguards are intentionally broad right now and may flag safe and routine coding, cybersecurity, or biology work"——常规的编程、安全测试、生物相关工作都可能被误标。个人知识库、归档自己管理的公众号、用自己的登录态做同步这类日常工作流，一句话里缺了"这是我自己的"这层上下文，就容易被按最坏情况解读。

### 核心哲学：消歧义，不是规避

> **本模式的唯一手段是"增加信息"，绝不是"减少信息"。**
> 让分类器看清楚这是正当请求，靠的是把所有权、授权、用途说清楚——不是靠替换敏感词、拆分表述、或任何让分类器"看不见"的手法。
> 这是**消歧义（disambiguation）**，不是**规避（evasion）**。分界线：消歧义之后，请求包含的事实信息只会更多、更具体；规避是事实信息不变或更少，只是换了一层包装。

本模式不是对抗分类器，而是**替用户把话说完整**——把用户心里清楚、但打字时习惯性省略的所有权/授权/用途信息，显式写出来。

### 误报诱因清单（四类）

1. **裸敏感词无上下文**：`cookie`、`token`、`credential`、登录态、抓取、injection、爆破这类词本身语义中性——技术上就是这么叫的，但脱离上下文单独出现时，容易被当成攻击意图的信号词。
   > 例："帮我处理一下这个 cookie" / "写段代码测试 injection"——没说目标系统是谁的、什么环境、什么目的，分类器倾向按最坏情况解读裸词。
2. **所有权不明**："这是谁的东西"没说清楚。同一个动词（抓、拉、同步）用在"自己的"和"别人的"资产上，风险性质完全不同。
   > 例："抓公众号文章"（谁的号？）vs. "归档我自己管理的公众号已发布文章"。
3. **授权不明**："凭什么能拿到"没说清楚。"我本人已登录的会话"和"想办法弄到的后台权限"是完全不同的两件事，但只写动作不写来源时，分类器分不出差别。
   > 例："拿到后台 cookie" vs. "从我本人已登录的浏览器会话导出本机凭据配置文件"。
4. **用途不明**："拿来干什么"没说清楚。"备份到个人知识库"和去向不明的"批量拉取"，后者天然更容易被联想成二次分发或未知目的。
   > 例："批量拉取历史数据" vs. "一次性备份到本地个人知识库"。

### 改写四原则

逐条对应四类诱因；四项都说清楚，分类器才有完整信息做判断。

1. **声明所有权**——写清楚"是谁的"：我的账号 / 我的 vault / 我自己经营或管理的公众号 / 我本人的浏览器会话。不写"这个号""那个网盘"这类不带归属的指代。
2. **声明授权**——写清楚"凭什么拿到"：本人登录态 / 官方 API / 已经获得的明确许可。不写"想办法弄到""想办法绕过"这类不说明来源的动作。
3. **声明用途**——写清楚"拿来干什么"：个人备份 / 知识管理 / 学习研究 / 本地同步，并说明不对外分发。
4. **精确术语**——把江湖黑话换成技术上准确、可审计的说法："偷 cookie"→"读取本机凭据配置文件"；"黑进去"→写出具体接口名；"爬 / 扒"→"通过官方 API 导出"；"搞定权限"→写清楚已获得的访问范围具体是什么。

四原则是同一件事的四个检查点：**这句话有没有说清楚"谁的、凭什么、干什么、用什么词"**。

### Before → After 样例库

覆盖 PKM 与开发日常最常见的七类场景。**注意：以下样例均假设所有权/授权信息已经客户确认属实——实际执行必须先完成流程步骤 2-4 的诊断与确认，不能直接照抄样例结论把"我自己的"写进未经确认的改写句。**

**样例 1 · 公众号归档**（缺：所有权、授权、用途）
- Before：`帮我把这个公众号的历史文章全部抓下来。`
- After：`帮我把我自己经营的这个公众号的历史已发布文章，用公众号后台的官方素材接口批量导出，备份进我的个人知识库，不对外分发。`
- 改动：补"我自己经营的"（所有权）；"抓"→"用官方素材接口导出"（授权+精确术语）；补"备份进个人知识库，不对外分发"（用途）。

**样例 2 · 微信读书同步**（缺：授权来源；术语黑话感）
- Before：`把我微信读书的 cookie 弄出来同步笔记。`
- After：`从我本人已登录的微信读书网页会话里，导出登录态对应的本机凭据配置文件，用于同步我自己账号下的划线笔记到本地知识库。`
- 改动："cookie 弄出来"→"导出登录态对应的本机凭据配置文件"（精确术语）；显式声明"我本人已登录""我自己账号"（授权+所有权）。

**样例 3 · 云盘批量扫描**（缺：所有权、用途）
- Before：`写个脚本把整个网盘扫一遍，把文件列表导出来。`
- After：`写个脚本扫描我自己的云盘账号，导出全部文件的目录清单，用于给我的个人资料库建索引。`
- 改动：补"我自己的...账号"（所有权）与"给个人资料库建索引"（用途），消除"扫全盘"听起来像穷举探测的歧义。

**样例 4 · 浏览器登录态导出**（缺：所有权+授权、用途、排除他人账号）
- Before：`把浏览器里的登录态导出来给脚本用。`
- After：`把我本机浏览器里、我本人已登录的会话导出为本地凭据文件，提供给我自己写的同步脚本读取，不用于访问他人账号。`
- 改动：明确"我本机""我本人已登录""不用于访问他人账号"，排除"获取他人登录态"的解读空间。

**样例 5 · 批量抓取历史数据**（缺：所有权/授权、用途）
- Before：`批量爬取这个平台过去三年的数据。`
- After：`通过官方 API，批量导出我自己账号在这个平台过去三年产生的数据，一次性备份到本地，不用于二次分发。`
- 改动："通过官方 API"（授权）+"我自己账号"（所有权）+"一次性备份到本地，不二次分发"（用途）；"爬取"→"导出"消解攻击性动词。

**样例 6 · 安全测试类表述**（缺：所有权/授权、用途）
- Before：`帮我写一段能触发 injection 的 payload 测试一下。`
- After：`帮我写一段 SQL injection 测试用例，用于我自己负责维护的这个测试环境做安全回归测试，不针对生产环境或他人系统。`
- 改动：声明"我自己负责维护的测试环境"（所有权+授权）、"安全回归测试"（用途）、"不针对生产环境或他人系统"（显式排除歧义）。

**样例 7 · 项目结构摸底（真实误伤案例还原）**（缺：归属；"摸清...结构"缺落点时易被泛化为信息收集行为）
- Before：`摸清这个项目的规范文档和某个技能目录的结构。`
- After：`阅读我自己维护的这个开源仓库里的规范文档，并查看其中某个技能目录的现有结构，目的是照规范新建一个技能。`
- 改动：补"我自己维护的开源仓库"（所有权）、"目的是照规范新建一个技能"（用途），让中性但抽象的动作有清晰落点。

### 模式 C 流程

> **本模式下的主干信息 = 所有权 + 授权**（它们直接决定红线判断），对应第一性准则里"必须先问、不能占位"的范畴；用途与精确术语相对次要，仅这两项缺失时才可参照占位符例外先行产出并标注待确认。

1. **接收原句**：可能还没发出，也可能是发出后被拦截 / 降级 / 追问的那句话。
2. **逐项诊断**：对照四类诱因检查——裸敏感词有没有上下文？所有权、授权、用途各说清楚了吗？
3. **红线检查（优先于改写）**：如果原句描述的操作本身就是未授权访问、他人账号、绕过访问控制——**停止改写**，直接告诉客户问题所在。
4. **只在诊断通过后改写**：把缺失项用客户确认属实的信息补全，同时把黑话/模糊动词换成精确术语。客户答不上来"这是谁的账号""有没有授权"，就是诊断没过——先问清楚，不代替客户下结论。
5. **输出改写版 + 说明**：可直接复制的完整句子 + 逐条"改了什么、为什么"，让客户自行核对改写后的表述是否符合事实。

### 红线声明（必须显著，优先于其他所有规则）

**本技能只用于消除对"正当请求"的误报，不用于帮助掩饰"确实有问题"的请求。**

以下情况**不改写，直接指出问题**：

- 请求涉及访问客户本人不拥有、没有合法授权的账号或系统（"这个号不是我的，但我想…"）。
- 请求涉及绕过访问控制、认证机制或平台风控（"怎么绕过验证码""怎么绕过登录限制"）。
- 请求涉及获取他人未公开的凭据、隐私数据，或未经许可批量获取他人数据用于二次分发、牟利。
- 客户明确表示所有权或授权声明"不完全属实"，只是想让请求"看起来"合规。

命中以上任意一条时，回复只说明问题所在——不给出改写版，也不做任何语言包装或美化。

**改写不能改变事实**：

- 改写只能显化客户在诊断环节确认过的真实信息，不能替客户编造"我自己的""已获授权"这类声明。
- 如果客户答不上来"这是谁的账号""有没有授权"，就是诊断没过，不能进入改写环节——应该先问清楚，而不是代替客户下结论。
- 判断标准：改写前后，事实性内容（谁拥有、谁授权、做什么用）必须完全一致；改变的只是"表达是否清晰完整"，不是"事实是什么"。

## 边界与限制（三模式通用）

- 本技能只产出提示词文本和说明，不代替执行提示词本身，也不保证目标 AI 一定产出理想结果——提示词是必要条件，不是充分条件。
- 模式 C 不保证改写后一定不被分类器拦截——分类器的最终判断不受本技能控制；本技能只保证把能提供的上下文尽量说清楚。
- 本技能不是越狱（jailbreak）工具，也不是提示注入（prompt injection）工具；任何模式下都不接受"帮我绕过审核 / 绕过安全限制"作为目标。
- 本技能不获取、不生成、不验证任何账号权限——所有权和授权是否属实，由客户自己负责，本技能只负责表达层面的诊断与产出。
- 写提示词阶段只处理客户主动提供的文本，不读取客户的实际账号、vault 或本机文件；执行阶段的一切动作以展示过的提示词内容和所选执行器的规范为准。
- 「产出并执行」不改变三模式的产出纪律：澄清、红线、回执一条不少；执行器的失败如实回传，不代为修饰。

## 完成后回执

执行完**必须**向用户输出（不要默默做完）：

1. **做了什么** — 一句话总结本次模式（A 新写 / B 优化 / C 防误伤）、澄清情况和产出结果；模式 C 命中红线时改为说明"本次不改写，原因是…"。
2. **文件变更** — 本技能不写文件、不做网络调用，此项固定回答"未改动文件（本技能仅输出文本诊断与产出结果）"。
3. **下一步** — 模式 A/B 提醒客户实际试跑产出的提示词、效果不符再拿回来走模式 B；模式 C 提醒客户核对改写句里的所有权/授权声明是否真实，不属实需重新诊断而不是直接使用。
