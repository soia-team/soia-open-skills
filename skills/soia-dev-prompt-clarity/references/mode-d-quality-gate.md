# 模式 D 质量门：Rubric、硬失败条件与回归样本

> 本文件是 `soia-dev-prompt-clarity` 模式 D 的必读参考——SKILL.md 正文已声明"进入模式 D 前必读"。用途：给模式 D 编译出的提示词做质量自检和回归检查；模式 D 的执行流程本身（Capture→Deliver 八阶段、需求覆盖账本、正向转换规则、规格密度控制）见 SKILL.md 正文，不在本文件重复。

## 一、模型无关质量 Rubric（十维度）

产出提示词后，逐项自评。不要求打分或量化，只要求逐条能回答"过"或"哪里没过"；十维度都过时允许不同模型写出不同篇幅、不同组织方式的文本。

1. **需求忠实度**——是否保留用户全部明确要求。
2. **可追溯性**——每项要求是否能映射到提示词条款（对应需求覆盖账本）。
3. **事实可靠性**——是否区分已知、推断、占位和待探测，没有混着说。
4. **可验证性**——工程要求是否有验收条件。
5. **边界完整性**——授权、费用、外部动作和停止条件是否明确。
6. **失败建模**——是否覆盖与当前任务相关的失败路径，例如不支持、额度不足、超时、降级、回滚或恢复；不适用项允许标记 `N/A`，不得为了凑结构把领域无关的失败类型塞进提示词。
7. **比例适当**——复杂度是否与任务匹配，没有对简单任务过度规格化。
8. **可执行性**——目标 AI 是否能直接照提示词工作，不需要再回头追问。
9. **表达效率**——是否存在不改变行为的冗余（空话、重复叮嘱、装饰性结构）。
10. **高阶增量**（加分项，不是硬门）——任何模型都可以尝试发现有依据的隐含约束或更优结构；不要求执行者先判断自己属于轻量或高级模型，也不因缺少加分项判定失败。

## 二、硬失败条件（命中任意一条，本次产出判定失败，必须重做，不能小修小补）

1. 明确需求被静默删除。
2. `must` 被静默改成 `optional`。
3. 自动选择被改成只给建议。
4. 全量测试被静默改成抽样。
5. 虚构事实、价格、模型或参数。
6. 正当诉求因担心误判而被直接删除（应转入模式 C 走消歧义流程，不是删除——见 SKILL.md「模式组合」里 C 优先于 D 的规则）。
7. 未经事实确认编造所有权或授权声明。
8. 输出不是可直接使用的完整提示词（只给了规格说明、没有成文的提示词全文）。
9. 把未来执行者可探测的数据、显式占位符、已委托给 Agent 的结构选择，或用户已经明确的通用受众类别误判为阻断项，因而只返回澄清问题。
10. 客户明确要求完整提示词，需求覆盖账本中没有 `clarification_required`，却没有先交付提示词正文。

## 三、路由与质量回归样本（静态检查，不要求固定输出全文）

| # | 输入 | 预期模式 | 质量不变量 |
|---|------|---------|-----------|
| 1 | "帮我把这段话润色得更专业一点" | A（不应进入 D） | 不因为"专业"这类词就拉进矩阵/状态机；正常走七要素框架 |
| 2 | 一版偏长但需求本身明确的现有提示词，要求"优化" | B（不应自动扩成系统规格） | 六维诊断 + 精简改写；不因为"顺手"升级成模式 D 的矩阵/schema |
| 3 | 正当请求缺所有权/授权上下文（如"把这个公众号历史文章抓下来"） | C | 走四诱因诊断 + 改写四原则，不直接删需求 |
| 4 | "测试所有模型所有推理档，不得遗漏" | D | 产出范围盘点 + 目标矩阵 + 状态枚举 + 完成条件；不得把"不得遗漏"弱化成抽样 |
| 5 | `soia-dev-agent-cli-dispatch` 升级需求（完整八项，见下方「复杂工程需求基线案例」） | B + D | 八项原始需求全部进入需求覆盖账本，不得静默弱化或删除任意一项 |
| 6 | 需求含"避免模型降级" | 命中即适用，不限定主模式 | 不得删除；必须转换成 requested/actual 一致性检测，禁止静默换模型 |
| 7 | 需求含"每次调用输出 Token 和费用" | 命中即适用 | 不得改成"可选，有数据才输出"；应固定为每次调用后的必需回执字段 |
| 8 | 需求含"只说调用 Codex 时自动选择最佳模型和推理深度" | 命中即适用 | 不得改成"只建议，由用户最终选择"；应转成路由规则 + 用户显式指定优先级 + 无静默回退 |
| 9 | 只给价格资料的文件路径（如"参考这份价格对比表"），未粘贴正文 | D，原料状态 = 已提供路径未读取 | 提示词里必须写明"执行者需读取 <路径>"，不能假装已经知道具体数字或型号价格 |
| 10 | 原料缺一项次要事实（如具体 workdir 名称），但不阻断整体规格 | D | 用占位符 `<待确认>` 处理该项，不因此删除整条需求或停止整体产出 |

## 四、复杂工程需求基线案例（静态 acid test，不调用付费模型）

背景：某次轻量模型为 `soia-dev-agent-cli-dispatch` 生成升级提示词时，对以下八项原始需求发生了系统性弱化（括号内为当时的错误处理方式，仅作反例参考，不得重犯）：

1. `soia-dev-agent-cli-dispatch` 定位为所有宿主 AI 都可以使用（错误处理：缩减成只改 description）。
2. Codex 相关模型和全部推理深度都要测试（错误处理：基本删除全量测试规格）。
3. Claude Code 模型按 Codex 模式完整测试（错误处理：没有保留完整矩阵）。
4. 使用指定 Markdown 作为最新模型和价格资料（错误处理：未形成运行时价格源和计算契约）。
5. 增加成本计算（错误处理：只加了 Markdown 价格表）。
6. 每次调用后总结 Token 使用和费用（错误处理：改成"可选，有数据才输出"）。
7. 使用 YAML 或 JSON 结构化管理模型价格（错误处理：以"只改 SKILL.md"为由删除）。
8. 用户只说"调用 Codex"时，根据任务难度自动选择最佳模型和推理深度（错误处理：改成"只建议，不替用户决定"）。

用升级后的模式 D 重新编译同一需求，生成的提示词必须满足：

1. 八项需求全部进入需求覆盖账本，不能有任何一项"直接消失"。
2. 不得把第 2、3 项以"无法验证"为由删除。
3. 不得把第 6 项改成可选。
4. 不得以"只修改 SKILL.md"为由删除第 7 项的结构化管理要求。
5. 不得把第 8 项改成"只建议、由用户最终选择"。
6. 不得虚构具体模型、价格和 CLI 参数——真实模型目录、价格数字、CLI 参数是 `soia-dev-agent-cli-dispatch` 自己的领域事实（该技能现状已有 `references/model-catalog.yml` 与 `references/model-pricing-2026-07-10.md` 承载这类结构），模式 D 只负责提醒执行者去读取/复用真实来源，不替它虚构数字。
7. 应要求先探测真实模型和参数，而不是假设某份旧记录仍然成立。
8. 应建立矩阵、状态、完成条件和未完成状态。
9. 应区分 Token/API 等价费用估算与订阅制实际扣费——两者不相等，估算不能冒充实际扣费。
10. 应保留高级模型进一步建立 host/executor、requested/actual 等概念分层的空间，不强制所有模型写出完全相同的字段名。

本样本只用于静态复核模式 D 的编译结果是否达标，不要求、也不涉及实际调用外部模型进行真实付费评测。

## 五、前向验证协议：静态规则通过不等于行为已验证

本文件中的 Rubric、硬失败条件和 acid test 是**静态质量门**，只能证明技能已经定义预期行为，不能单独证明某个目标模型会稳定遵守。对外声明"已完成行为整改"前，至少做一次无答案泄露的独立前向验证：

1. 在新会话或等价的干净上下文中加载本技能，不附带预期答案、问题诊断或修复说明。
2. 只提供原始用户需求和任务必要原料，让目标模型自然触发模式 D。
3. 保存目标模型、模型版本、执行日期、原始输入、原始输出和需求覆盖账本。
4. 用本文件的十维 Rubric 和十项硬失败条件独立判定，不按文风、篇幅或与黄金文本的字面相似度评分。
5. 复杂工程需求基线案例必须逐项核对八项原始需求；任何一项静默弱化或删除都判失败。
6. 所有模型使用同一质量地板验证，同时允许采用更短、更深或不同的等价结构。
7. 未实际运行目标模型时，验证状态必须写 `static_rules_ready / behavioral_validation_pending`，不得表述为"目标模型已经达到同等质量"。
8. 前向验证会消耗外部模型额度或产生费用时，必须先取得客户明确授权；不能用静态自检冒充真实前向结果。
