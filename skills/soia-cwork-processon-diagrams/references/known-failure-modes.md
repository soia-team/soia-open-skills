# ProcessOn 盘点与归档失败模式

这份参考记录把真实执行中容易“看起来完成、实际上不完整”的模式固化为门禁。执行技能时只在相关场景读取，不要把它当作用户报告模板。

## 1. 顶层页面不等于全量

**错误表现**：只读取团队根目录或一级目录，就按页面上看到的文件数声称完成；旧浏览会话中的数字也被当作当前事实。

**纠偏规则**：

- “全量”“全部文件”“所有子节点”一律递归到叶子目录。
- 每个父目录声明的 `folders` 都必须进入队列；只有实际打开并读取文件列表，才可以加入 `visited_paths`。
- 用 `discovered_paths - visited_paths - blocked_paths` 计算差集；差集非空只能报告部分完成。
- 没有运行包的旧会话只能作为 `legacy observation`，不能合并进当前 checkpoint 的完成统计。

## 2. 浏览器会话不是中间数据库

**错误表现**：目录队列、已读结果或多 Agent 的结果只留在会话上下文，浏览器刷新/线程切换后无法恢复；多个 Agent 同时写 checkpoint 导致批次编号或统计竞争。

**纠偏规则**：

- 开始即建立 XDG state 运行包；每 3—6 个目录生成不可变批次，先 `record` 成功再继续浏览。
- 多 Agent 可以并行只读探查，但同一运行包只能有一个状态写入者；批次必须由主控串行合并。
- 中断恢复只读 `pending_paths`，不要凭记忆或重新扫一级目录。
- 最终结论必须由 `audit` 重放批次得出，不能用页面提示、模型记忆或 checkpoint 单独证明。

## 3. 验证码要按“当前视口可见且阻断”判定

**错误表现**：页面 DOM 中存在预加载的安全验证 iframe，就声称用户看到了滑块；或为了继续任务尝试模拟拖动、调用内部接口。

**纠偏规则**：只有验证码/遮罩与当前视口相交、真实可见并阻断目标操作时才停止并请用户接管。`display:none`、零尺寸、位于视口外的 iframe 不是验证码弹窗。不得绕过、模拟或逆向验证。

## 4. 登录凭据属于浏览器，不属于 Agent

**错误表现**：让用户把密码发到聊天中，或尝试通过 CDP 参数、Chrome 扩展、环境变量、Cookie/Local Storage 读取登录态。

**纠偏规则**：Agent 只打开官方登录页并等待；用户在当前浏览器手动输入用户名、密码、短信码和滑块。Agent 只读取登录完成后的页面，不读取输入值、Cookie、Token 或 profile。

## 5. 盘点完成不等于资产归档完成

**错误表现**：目录审计通过后，报告直接写“知识库归档完成”，但实际上没有下载任何 VSDX/XMind/POS 文件。

**纠偏规则**：明确分离两个阶段：

1. `inventory`: 目录、文件、类型和差集已审计；
2. `artifact_archive`: 源文件下载、实际格式校验、SHA-256、manifest、draw.io 转换和预览生成。

目录阶段完成后先运行 `build_processon_archive_plan.py`，生成每个文件的稳定 `artifact_id`、默认格式和待确认队列。计划中的 `ready_for_known_artifacts=true` 允许已确认类型分阶段下载；`ready_for_archive=true` 只有在 unknown 清零后才代表全量条目均可归档。每个下载文件仍需经过 `finalize_processon_download.py`，不能用浏览器 Toast 代替落盘证据。

### 增量不等于一次局部扫描

**错误表现**：只检查一级目录、按修改时间猜变更，或把没有 ID 的同名文件判断为“移动”；不完整的当前扫描直接把历史条目写成“已删除”。

**纠偏规则**：先完成并审计新的全量 checkpoint，再用 `diff_processon_inventory.py` 与同范围历史 checkpoint 比较。它只产生本地快照差分，明确写出两份 SHA-256；当前或历史任一快照有 pending/blocked 都 fail closed。没有 `remote_id/id` 时只保留新增和移除候选，绝不声称移动；同目录完全相同的无 ID 条目隔离为不可判定项。候选也不会自动删除源文件或已归档文件。

## 6. 类型选择必须有证据

**错误表现**：根据颜色、文件名或单一图标猜流程图/思维导图；把未知项静默下载为 VSDX。

**纠偏规则**：

- 流程图默认 `VSDX`，思维导图默认 `XMind`。
- 多画布流程图选择“导出全部画布 (.vsdx)”；默认格式不可用或会员/权限阻断时回退 POS，并记录原请求、实际格式和原因。
- 无法从卡片、浏览视图或菜单确认类型的条目保持 `unknown`，进入人工确认队列，不自动下载。
- 建议保留 POS/POSF/POSM 作为 ProcessOn 原生结构备份，但“建议”不等于已经下载。

## 7. 同名文件和目录必须保留定位信息

**错误表现**：用标题作为唯一文件名，导致不同目录或同目录同名条目互相覆盖；下载成功但无法回溯来源。

**纠偏规则**：计划项必须包含完整逻辑目录、标题、类型、所有者、远端更新时间，以及可见时的 `remote_id/source_url`。归档使用稳定 `artifact_id` 和原目录层级；同目录同名且没有远端 ID 时，目录名附加稳定 `artifact_id` 前缀（如 `未命名文件--8ba9f60f/`），不能只依赖浏览器的 `(1)` 后缀，禁止静默覆盖。

## 8. 失败必须可交接

每个受阻项都要写入：来源目录、目标文件、阶段（浏览/导出/下载/校验/转换）、错误证据、下一步人工动作。不能把“尚未开始”“会员限制”“验证码阻断”“未知类型”混在一个失败数里，也不能用“预计完成”掩盖未执行阶段。

正式资产归档使用 `processon_archive_state.py` 维护唯一可恢复队列：`record` 只接受已经落盘、结构和 finalizer manifest 一致的文件；`mark` 分开保存 failed 与 blocked，并可用 `--evidence-file` 把诊断文件复制到运行包；`audit` 从实际文件与阻断证据重算 SHA-256。知识库 Markdown 是冻结摘要，不反向充当运行队列。

## 9. 虚拟列表定位超时不等于文件不存在

**错误表现**：目标标题已在全量 checkpoint 中，但页面定位超时，于是把它标成删除、权限不足或下载失败。

**纠偏规则**：先确认当前逻辑目录，再滚动虚拟列表让目标条目进入视口，重新读取页面快照后再操作。连续两次滚动无新增只用于目录盘点收尾，不可替代逐项下载时的目标可见性检查。

## 10. Markdown 能下载不代表思维导图原生归档完成

**错误表现**：XMind/POS/POSM 点击后无文件，随后 Markdown 下载成功，就把 artifact 记成已完成或把 Markdown 改名成 `.xmind`。

**纠偏规则**：先在列表页和官方编辑器分别验证 XMind，再回退 POS/POSM；所有原生格式都无产物但 Markdown 可下载时，Markdown 只证明登录态、文件权限和下载通道正常。用 `mark --outcome blocked --evidence-file <markdown>` 保存可重试阻断，绝不调用 `record`，更不得改扩展名冒充原生包。

## 11. 菜单点击完成不等于异步导出完成

**错误表现**：按固定等待时间连续点多份 VSDX；ProcessOn 前一份仍在生成时，Agent 已切换到下一条。最终文件内容属于前一项，文件名却取了后续当前选中标题，出现同名不同 SHA、连续 `(1)` 后缀或标题与页面文字冲突。

**纠偏规则**：同一标签严格单文件串行。每份都以“新增文件出现、临时下载结束、大小连续稳定、结构校验通过、必要时页面文字能唯一对应 artifact”为完成信号，完成 `finalize` 与 `record` 后才切换条目。短时下载事件超时只代表事件未捕获，不代表导出失败；此时等待并检查下载目录，禁止立即重复点击。无法通过 VSDX 页面文字唯一反证来源的文件不得改名或归档，artifact 保持 pending。

## 12. 宿主浏览器工具会干扰客户并制造孤儿标签

**错误表现**：批量循环依赖 Codex/Claude 的 browser、computer-use 或 Chrome 扩展，在客户正在使用的主 Chrome 中逐份打开页面；即使平时只保留一个源页面，也会持续抢焦点。任务中断时当前页遗留，其他 AI 又无法复用这套执行链。

**纠偏规则**：正式批量只用 `processon_browser_runner.py` 的技能专用 profile，首次登录走独立 headed 窗口，后续默认 headless。启动时收拢为一个父页面；临时新页必须写成嵌套 `popup`，在 `finally` 中关闭；整次 context 在成功、异常、超时、Ctrl-C 和 SIGTERM 后全部关闭。回执中 scoped 打开/关闭数不等或退出关闭数为 0 时，本批视为中断待核查。宿主浏览器能力只可在客户明确同意后做一次性诊断，不能回退成批量主路线。

## 13. “多开浏览器”不能跳过交叉串件验证

**错误表现**：为了提速直接启动多个主 Chrome、多个无锁 orchestrator，或只看到并发下载的文件名正确，就假定文件内容也正确；每份打开新标签但不关闭，继续干扰客户操作。

**纠偏规则**：只在一个技能专用 persistent context 中固定复用 1–3 个 headless worker 页；同一 worker 严格串行，源图 popup 必须由发起点击的 worker 页用 `expect_popup` 独占，并在 `finally` 关闭；批次退出关闭所有 worker 和 context。默认串行；两路/三路分别需要当前计划的真实并发 proof，proof 必须解析 VSDX/XMind 内部文字反证没有串件。每个 artifact 使用独立 staging 子目录，禁止平铺到个人下载目录后依赖 `(1)` 后缀。下载可并发，但 finalize、`metadata.yml`、source-links、progress mirror、`record` 和 `audit` 只能由持有全局 lock 的单 writer 执行。collision-risk 项在串行和并发模式下都排除，直到取得行级稳定身份；发现内容语义不匹配、相同 SHA 或关闭数不对时，整项保持 pending 并降级并发度。

## 14. 团队 URL 不保证回到团队根目录

**错误表现**：ProcessOn 会记住团队空间上次停留的子目录；重新打开相同 team URL 后，页面仍在旧 breadcrumb。脚本把 URL 当根目录，随后在当前文件列表中找一级目录并超时。

**纠偏规则**：每个 job 导航前读取可见 breadcrumb，校验首项与计划根一致，再通过固定 breadcrumb 根节点回到团队根；逐层导航时只接受 `file_list_item` 内唯一可见的文件夹行，不按页面任意同名文字点击。breadcrumb 无法归一、目录行不唯一或最后路径不符时保持 pending，不增加等待冒充修复。

## 15. 版本号不能粘连中文语义信号

**错误表现**：标题 `磐石4.0短信系统部署架构图` 被拆成 `磐石4` 和 `0短信`，而 VSDX 内部自然写成 `磐石-短信(4.0)`，导致来源正确的文件被误判为语义不匹配。

**纠偏规则**：生成标题信号前先把 `4.0`、`3.0.1` 等点分版本号替换为分隔符，再提取 `磐石`、`短信` 等相邻中文业务词；仍须至少命中一个有区分度的信号，不能因修复误报而取消语义门禁。
