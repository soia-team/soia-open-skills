## 先判断动作风险

| 动作 | 默认策略 | 必要证据 |
|---|---|---|
| `ls/ll`、扫描、SHA1 读取 | 可直接做 | 命令、driveId、扫描范围 |
| 改名/同盘移动 | 先落账本；在批准方案范围内执行 | from/to、操作回显、独立终态 `ll` |
| 删除/覆盖上传 | 必须有用户明确授权 | 删除/覆盖清单、旧实体证据、回收站或新实体复核 |
| 跨盘移动 | 视为复制+删除，单独批准 | 源/目标 SHA1、bytes、旧链接更新 |

每条云盘命令显式传 `--driveId`。不要依赖 CLI 当前盘位；个人盘和备份盘同名路径会让“看起来成功”落到错误位置。

执行清单先写入 `${XDG_STATE_HOME:-$HOME/.local/state}/soia-pkm-alipan-curator/moves/`。账本记录计划与验证，不记录 token/cookie。

## 扫描与路径异常

### `LIST_FAIL` 不是空目录

症状：DFS/`ll` 返回“指定目录不存在”，常见于空格、括号、特殊字符、短暂状态或 CLI 路径解析问题。

处理：

1. 把该路径写入 `.errors`/回执并标为保护例外。
2. 换 `file_id` 或父目录逐层 `ll` 重新读取。
3. 未取得“目录存在且子项数为 0”的独立证据前，不进入空目录删除清单。
4. 索引可以标注扫描缺口，但不能把缺口算成 0。

### `ll` 表格解析必须保留文件名中的连续空格

症状：父目录明明能看到资源，执行器却报 `SOURCE_NOT_LISTED`；计划路径与云盘名称肉眼近似，但文件名中的两个或更多连续空格被解析器压成了一个。

处理：

1. `aliyunpan ll` 的名称位于前九个固定字段之后；解析时只切前九列，把剩余文本原样视为名称，不能按 `\s{2,}` 拆整行。
2. 计划和结果账本必须保留原始名称；不要为“匹配成功”手工压缩空格或改写云端实体。
3. 已记为 `skipped/SOURCE_NOT_LISTED` 的动作在修复解析器后从断点重跑；只有独立终态 `ll` 得到 `verified` 才闭环。
4. 回归测试至少覆盖目录名和文件名中的连续空格，防止扫描器与执行器采用不同解析规则。

### 不要为盘点下载整套资源

目录结构、文件数、大小、SHA1、扩展名和 file_id 足以完成大多数盘点/整理。只有判断音轨、字幕、损坏或内容真伪时，才下载最小样本并用 `ffprobe`/播放器验证。用户未要求时，不全量下载媒体。

### 扫描根必须显式

增量扫描要提供 `--roots`，把本次真实根分区的 file_id 写入。否则某条深层扫描失败后，孤立记录可能被错误提升为新的顶层分区。

## 编号与目录结构

### 只给业务层编号

需要稳定排序的是分区、课程和教学材料类别。`images`、`banner`、`scripts`、`assets`、离线网站相对路径等技术依赖树保持原名；全量编号会破坏依赖，也会制造冗长的累计编号链。

- 编号是可选排序工具；只有用户确认使用时，才由本次方案声明类目层步长和格式。
- 课程内供用户选择的“正课/练习/解析/答疑/其他”是业务导航目录，不因层级较深就自动排除；单集文件和技术依赖目录才保持原名。
- 顺序层如使用 `01_`、`02_`，只写本层编号，不把父级编号重复拼进名称。
- 待核/存档区的名称和编号不是公共固定值，但在同一用户库内应保持一致。
- 无编号旧目录不能只在索引里“藏起来”；如果属于业务层，应整理实体目录后再刷新索引。

### 长系列超过单夹上限

- 先确认用户本次选择的上限；公共 skill 不默认 20、50 或其他数字。
- 文件名有稳定序号时按范围分组，文件名有多个重启序号的系列标记时先按系列拆开。
- 分组目录属于浏览导航层，可按本次编号合同加前缀；集文件保留原名和原编号。
- 一个系列需要 10 组以上时，前缀宽度按最大编号统一扩展，例如 `010/020/.../100`；不能让 `10/20/.../90/100` 在名称排序中错位。规划 dry-run 后必须直接比较分组名列表与其字典序。
- 源编号缺失只报告缺集，不能为了“连续”重编号或创建占位文件。
- 终态用 `chunk_layers` 审计：根目录无散文件、每组非空、每组主项目数不超过 `max_items`；视频带字幕/XML、音频带封面等侧车时，用 `count_pattern` 只计主媒体，侧车仍必须跟随入组。技术依赖目录逐名写入 `exclude`，未匹配且未豁免的同级目录必须报错。
- 同一源父目录需要移动大量实体时，可使用 `apply_reclass_bulk.py --batch-size 20`；计划必须把同源同目标 action 连续排列。批量只减少 CLI 往返，不降低证据粒度：批前/批后都读取源与目标，并为每个 action 单独写 `verified/failed`。
- 多个超长系列先用 `plan_series_chunks.py` 从逐文件扫描和本次规则生成计划，不在临时目录另写 builder。规则要逐系列声明主媒体、可选集号、侧车和保护项；未匹配直属文件默认阻断，主控审核报告后才执行。
- 再用 `flat_series_discovery` 做全区第二遍发现，专门找未进入 `chunk_layers` 的超限平铺目录。自然月份等稳定语义桶如需保留，必须在本次合同中用路径正则显式排除；公共技能不内置例外。
- 发现审计必须喂逐文件终态扫描，不能使用 `agg_files` 聚合或 `no-descend`。运行时把扫描错误 sidecar 传给 `--scan-errors`；缺失、非空或路径写错都失败。路径/文件规则零匹配也默认失败，只有已独立确认确实为空时才设 `allow_empty=true`；仅在另有完整性证据时才使用 `--allow-missing-scan-errors`。

### 根级“先看这里”不能漏审

`guide_layers` 是“遍历某父目录下匹配的子分类”，不会自动检查父目录本身。若整个学习库根也要求统一入口，必须另写 `required_guides`，并提供 `file_pattern` 验证正式说明文件存在；文件默认至少 1 字节，可按产物设置更高的 `min_bytes`。只建空目录或零字节占位不算完成；规则匹配不到任何分类也会失败，确认该层确实为空时才显式设置 `allow_empty=true`。

### “可直达”必须是可点击链接，上传成功必须有实盘哈希

- 关键 Excel/说明上传后重新扫描或独立 `ll`，把最终路径、字节数、SHA1 和可选 file_id 写入 `required_artifacts`；命令回显、旧 manifest 或本地哈希单独都不算远端成功。
- 资源地图把根入口、重点分类/课程的最终 file_id 写入 `resource_maps.required_ids`，并显式提供本次云盘 `url_prefix`。审计只认 Markdown 链接，不认正文中的“可直达”文字或裸 URL。
- 若地图链接来自移动前的旧 file_id，或回执抄错 SHA1，结构审计必须失败并以独立实盘结果纠正，不能为了让合同通过而改实盘证据。

### 去套娃前先判技术依赖

“目录里只有一个子目录”不等于一定可上提。先检查 HTML、播放列表、字幕、脚本或配置是否引用相对路径；技术包整体保留。普通业务容器才按总数守恒上提并删空壳。

## 删除、空目录与去重

### 空目录删除

空目录必须有独立 DFS/`ll` 证据。删除叶目录后可能出现新的空父目录；是否继续删除父目录属于范围扩展，未授权就保留并报告。

回执同时报告：

- 删除操作次数；
- 删除的文件/目录实体数；
- 因级联变空但未在批准范围内删除的父目录。

不要把“9 次删除操作”写成“删除 9 个文件”，口径要清楚。

### 重复目录删除

整夹去重需三项全部闭合：

1. 相对路径集合一致；
2. 文件数一致；
3. 每个相对路径的 SHA1 一致。

同名、同大小或总大小接近都不足以删除。同名但 SHA1 不同的版本移入用户确认的待核区，或保留两份并说明差异。

部分重复时先求并集：删除已证实副本，把独有文件移动到保留侧，再次核对总数与 SHA1，最后才删除空容器。容器删不掉是有剩余项的信号，不要强删。

### 广告与说明文件

广告特征只能生成候选清单，不能绕过删除授权。用户若已明确批准某一规则和范围，可按该规则批量执行；否则先展示名称、位置、数量和可恢复性。

## file_id、链接与覆盖上传

### 同盘移动与覆盖上传不同

- 同盘 `mv/rename`：通常保留 `file_id`，已有目录深链继续有效。
- 上传并覆盖同名文件：旧文件进入回收站，新文件可能得到新的 `file_id`。
- 跨盘移动：按复制+删除理解，file_id 和链接通常变化。

所以“文件名没变”不能证明链接没变。覆盖前后都要记录并复核 ID/SHA1/bytes。

### 深链格式

使用：

```text
https://www.alipan.com/drive/file/all/backup/<40位file_id>
```

`drive/folder/<id>` 可能回到首页，不作为验收通过的链接。

链接更新后，全库搜索旧 file_id。当前资源地图、执行卡和导航表应为 0；历史审计回执可以保留旧 ID 作为证据。

## 索引与统计

### 先上传导航，再终态扫描

家庭导航/总索引本身也是云盘文件。正确顺序是：整理终态 → 生成导航 → 上传并复核 → 终态扫描 → OB 地图/全文检索 → 总 Excel。顺序反过来，索引天然少算新上传文件。

### 物理统计与逻辑大纲分开

同一路径字符串可能出现多个物理实体记录：

- 目录大纲可按逻辑路径去重，避免重复展示；
- 全盘目录数、文件数、体量和全文检索必须按原始扫描记录统计；
- 总 Excel 必须区分“物理总文件数”和“已展开明细数”。

### 增量拼接必须由生成器完成

局部重扫时使用 `gen_catalog.py --merge-existing ... --roots ... --search-dir ...`。一次增量输入只能包含一个明确根分区；只替换该分区正文和统计行，再按差额更新全盘总数，同时刷新同名全文检索文件。

不要手工复制 Markdown 表格。表格行之间多一个空行会让 Obsidian 显示断裂；生成器负责规范化。

## 并发与交接

- 只并行互不相交的云盘分区。
- 同一云盘分支、同一总览、同一全文检索文件、同一总 Excel 均只能有一个写入者。
- 公共目标目录由主控预建；跨区移动归属唯一执行者。
- 所有导航上传完成后做一次全区/全盘终态扫描，过程计数不能冒充最终计数。
- 接手其他 agent 的任务时，读账本和回执，但对删除、统计、链接做独立复核。
- 特大模块的中间证据统一放 XDG state 下的 `runs/<run-id>/`，不放进某个 AI 的 workspace。交接先给运行包路径，接手者先运行 `audit_run_bundle.py`，再按 `run.json` 读取初始扫描、焦点内容审计、计划/结果和终态证据；不要靠聊天摘要猜进度。
