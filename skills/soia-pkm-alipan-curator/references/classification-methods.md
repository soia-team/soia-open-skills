# 资源分类方法论与领域示例

> 用于深度整理前的分类设计与审计。本文提供决策方法，不提供任何用户的固定目录树。教育、编程、读书等结构只是示例；必须根据本次盘点证据和用户主要查找方式重新推导。

## 一、先真读再分类

文件夹名、宣传标题和扩展名只能帮助抽样，不能直接证明内容。先做全量结构盘点，再按资源包抽取可复核证据。

### 1.1 文档与电子书

至少读取：

1. 封面或标题页；
2. 目录；
3. 前言或第一章；
4. 中间随机一节；
5. 结尾、附录或版权信息。

记录真实主题、目标读者、内容形态、版本与完整性。扫描 PDF 无文本层时，先对少量代表页 OCR；不要只凭文件名归类。

### 1.2 视频与音频

先读文件清单、字幕、讲义和媒体元数据；用 `ffprobe` 核时长、音视频流、语言标签和编码信息。仍无法确认主题或受众时，只下载最小样本，在开头、中段、结尾各抽一处。没有播放或转写证据时写“未验证”，不要把课程名当作内容结论。

### 1.3 课程包与技术包

视频、讲义、代码、字幕和素材如果共同组成一门课程，应保持为一个资源包。不要为了按媒介分类拆散课程，也不要改动 `assets`、`images`、`scripts`、字幕等相对依赖树。

### 1.4 抽样边界

- 每个叶级资源包至少抽 3 个代表项；文件少于 3 个时全读。
- 同构系列可抽“首个 + 中间 + 最后一个”，但要先证明命名和内部结构同构。
- 名不符实、混合合集、疑似重复、可执行文件和高价值资源必须单独检查，不套用抽样结论。
- 把未读、读取失败和内容证据不足的条目列入 `unclear`，不得为了凑齐分类而强判。

## 二、选择主分类轴

分类的目标是让用户更快回答“它在哪里、该不该用、下一份新资源放哪”。同一层只使用一个主轴，其他维度放到下一层、索引字段或筛选列。

常见分类轴：

| 分类轴 | 适合回答 | 常见字段 |
|---|---|---|
| 主题/领域 | 我想找某个主题 | 商业、历史、语言、后端、健康 |
| 使用目的 | 我想完成什么 | 学习、参考、教学、创作、娱乐 |
| 受众/角色 | 谁来使用 | 学习者、照护者、教师、从业者 |
| 阶段/难度 | 先学什么、后学什么 | 入门、进阶、专项、项目实战 |
| 生命周期 | 现在是否可用 | 在用、待核、存档、待删除 |
| 媒介 | 用什么方式消费 | 书籍、视频、音频、软件 |
| 来源/系列 | 属于哪套产品 | 作者、机构、课程系列 |

选择顺序：

1. 写出用户最常问的 3 个查找问题；
2. 选择能回答最多问题、未来新增资源也稳定的轴作为本层主轴；
3. 检查同一层是否混入其他轴，例如“经济学 / 视频课 / 待整理”混合了主题、媒介和状态；
4. 将次要维度改为子目录、标签、Excel 列或馆藏卡字段；
5. 用全部条目试归类，重叠项和无处可放项回到规则层修正。

媒介通常不是一级分类。只有当用户明确按播放设备、版权形态或存储策略管理，且“找视频/找书”确实是主要入口时，才把媒介提升为主轴。

## 三、分类方案验收

对候选分类从五个维度各打 0-2 分，并写一句证据：

| 维度 | 0 分 | 1 分 | 2 分 |
|---|---|---|---|
| 覆盖 | 大量条目无处可放 | 少量待核 | 绝大多数可稳定归类 |
| 互斥 | 同一条可落多个类 | 依赖补充规则 | 按规则只有一个首选位置 |
| 可查找 | 用户仍需逐夹翻 | 部分问题可回答 | 主要查找问题可直接回答 |
| 可扩展 | 新资源会迫使重构 | 可有限扩展 | 新主题可插入且规则不变 |
| 迁移风险 | 要拆技术包或大范围跨区 | 可逆移动较多 | 主要是同盘改名/移动 |

总分低或任一项为 0 时，不进入执行。分数只帮助比较方案，不替代用户裁定。

分类合规审计还要检查：

- 是否把个人偏好写成公共默认；
- 是否按可识别个人身份命名；
- 是否拆散课程、软件或离线网站依赖；
- 是否把未验证内容伪装成确定结论；
- 是否仅在索引中隐藏问题，而实体目录仍然混乱；
- 是否存在同义类、交叉类或万能“其他”。

## 四、何时改分区边界

出现以下信号时，允许重命名、拆分或新增一级分区，但必须先给用户看影响范围：

- 分区名称只描述媒介，但实存资源包含多种媒介；
- 大量资源需要靠“其他/合集/暂存”才能容纳；
- 同一资源在两个一级分区重复出现或索引归属不一致；
- 新增资源总要跨越现有边界；
- 相邻分区的判断规则无法写成一句互斥规则。

先修正分区定义，再整理内部目录。不要为了保住旧名字制造更多例外。

### 4.1 分区边界必须写成互斥判断句

相邻分区不能只靠名字“感觉不同”。方案必须给每对容易混淆的分区写一句可执行判断：同一资源按这句话只能落一个首选位置。若用户同时维护课程区与书库，可以选择以下边界，但必须由本次实存内容与用户查找习惯确认：

- 独立电子书、独立读书笔记和纯书集归书库；
- 视频、讲义、字幕、代码或素材共同服务一套学习目标时，整包归课程区；
- 课程包内附带的 PDF 不因扩展名被拆进书库；
- 主题是分区内的分类轴，媒介只用于判断“独立出版物还是课程组成部分”。

这不是公共默认目录树，而是一种避免“同类书散落两区、课程又被按格式拆散”的边界写法。若用户主要按主题查找且不维护独立书库，则不要强行套用。

## 五、命名与编号

- 编号是可选的排序工具，不是所有用户的强制分类法。
- 用户选择编号时，步长、格式和适用层级由本次方案声明；只给业务分类层编号。
- `90_` 不是固定含义，也不是必建目录。需要待核/存档区时，使用用户确认的名称和编号。
- 每层只写本层编号，不把父级编号连缀进名称。
- “业务层”按用户是否需要在这里做选择判断，不按第几级目录判断。课程内部若并列“正课/练习/解析/答疑/其他”等入口，它们仍是业务导航层；若本次编号合同覆盖该课程，就应统一编号。
- 单集/单课文件、只服务播放顺序的章节文件、技术依赖树和出版物原始卷册通常保持原名。
- 方案必须列出“编号合同”：父目录、匹配格式、需要编号的直接子目录类型、排除项。执行后重新扫描合同中的每个父目录，漏号数必须为 0。
- 索引生成器可以折叠无编号素材树，但不能把业务目录漏号隐藏成“显示正常”。实体目录未闭环时，索引不得作为完成证据。

### 5.1 长系列分组合同

用户按集查找或网盘单页过长时，为课程、播客、视频、音频或卷册设置单夹上限。上限不是公共常量，由用户需求、客户端浏览体验和资源原有结构共同决定：

1. 未超过上限的完整系列可保持平铺，不为整齐强拆；超过上限才分组。
2. 有可靠原序号时按序号范围分组；存在两个各自从 01 开始的子系列时，先按子系列拆开，再分别按上限分段。
3. 无可靠序号时按内容阶段或模块分组，不按修改时间猜顺序。
4. 缺集只在回执中记录，不补空文件、不改后续原编号；附件跟随对应主课所在分组。
5. 合同声明 `parent / child_pattern / max_items`；确需保留的同级技术目录放入显式 `exclude`，不允许用通配豁免。终态要求系列根无散文件、未声明同级目录为 0、分组非空、每组直接文件数不超过上限。
6. 已声明的 `chunk_layers` 只能证明“知道的系列”整理正确，不能证明全区没有漏项。终态再用 `flat_series_discovery` 扫描本次范围内所有直接文件目录；`root / max_items / path_pattern / file_pattern / exclude_path_patterns` 全部由本次方案给出。自然月份、原出版卷册等稳定语义桶可显式例外，但不能靠技能内置名称静默跳过。发现审计只接受逐文件终态扫描；`agg_files` 聚合行、非空或缺失的扫描错误证明、路径零匹配、文件规则零匹配都必须失败，确知空范围时才显式传 `allow_empty=true`。

### 5.2 导览合同

当用户选择在学习分区设置“先看这里”时，导览也必须进入方案合同：导览目录名、适用层级、每个导览内的正式文件名和直达链接验收方式。分区根本身需要统一入口时，用 `required_guides` 单独声明根级导览；同级分类批量覆盖用 `guide_layers`。两者都可用 `file_pattern` 要求正式说明文件实际存在，并用 `min_bytes` 设置最低字节数（默认 1）；不能把空导览目录或零字节占位算作完成。`guide_layers` 匹配不到任何分类也默认失败，只有确认该层确实为空时才显式设置 `allow_empty=true`。只给一部分同级分类生成导览会形成新的结构漂移；终态要求应覆盖的分类全部存在、云端文件可打开，并被最后一次扫描与索引收录。

### 5.3 不确定项的复核区

若用户显式指定待确认/存档目标，可把 `unclear` 项隔离到该目标方便人工确认。按以下规则执行：

1. 每项记录 `source / reason / target / status`；
2. 独立文件可逐文件归档，课程或技术包整包归档；
3. 目标按来源或不确定原因分组，避免一个无上下文的大杂堆；
4. 完成时每项 `status=verified`，并验证目标位于用户指定复核根下且终态扫描确实包含该文件；
5. 公共 skill 不默认创建任何固定编号或名称的复核区。

## 六、领域示例

### 6.1 教育资源

如果学习者直用内容、照护者学习资料和机构教研资料混在一起，受众/角色可以作为主轴：

```text
学习者直接使用
照护者与家庭教育
教师与课程设计
待核
```

如果资源全都面向同一受众，则优先按学科、主题或发展阶段分类，不要机械套“受众三分”。教案里的音乐、图片和讲义属于同一课程包，不按媒介拆开。

### 6.2 编程与技术学习

通常先按能力领域分类，例如语言与基础、后端、数据、AI、云原生、工程工具，再在类内按入门/进阶/项目或课程系列组织。一本书、视频、源码和课件如果服务同一课程，应放在同一课程包；“视频”不是比“后端”更高的分类层。

含多种技术方向的“大合集”必须逐套盘点。不能仅因为文件名含 Java、Go 或 AI 就拆走内部章节；先识别课程根，再移动整个课程根。

### 6.3 读书与个人学习

纯电子书馆可以按主题、作者或阅读状态分类。若同一区同时有书籍、讲座、训练营、音频节目和实操课程，“阅读”已不足以描述范围，应考虑改成“个人学习”“通识学习”等目的型名称。

这类库通常按主题作为主轴，例如商业管理、经济社会、人文历史、表达创作、家庭与健康；书籍/视频/音频作为媒介字段。具体类名必须由实存内容推导，不能把示例照抄成默认目录。

### 6.4 内容创作与职业能力

摄影、剪辑、写作、演讲、新媒体运营既可能属于个人通识，也可能属于职业技能。判断依据是用户主要用途和相邻分区边界，不是文件格式。若已有职业/技术成长分区，应优先检查是否合并；边界不清时先保留原位并列待裁定项。

## 七、审计表模板

分类方案必须覆盖全部资源根，不只列疑问项：

| 路径/资源根 | 实读内容摘要 | 主要受众 | 主题/领域 | 媒介组成 | 完整性/版本 | 证据 | 置信度 | 建议位置 | 动作 |
|---|---|---|---|---|---|---|---|---|---|
| `<resource-a>` | `<从目录与样本提炼>` | `<audience>` | `<domain>` | `<formats>` | `<status>` | `<页码/文件/时间点>` | 高/中/低 | `<target>` | 保留/改名/移动/待核 |

方案文档另附：

1. 当前分区定义与问题；
2. 候选主轴及评分；
3. 目标结构与逐项映射；
4. 待用户裁定项；
5. 执行批次、回滚与索引刷新范围。
6. 编号合同、导览合同和不确定项复核根（若用户启用）。
