# 图书馆建法与整理方法论（library-method）

> 来源：2026-07-01~07 阿里云盘整理战役沉淀。补充 SKILL.md 未展开的"怎么把云盘变成图书馆"和"怎么定分类方案"的具体做法，含可复用模板与判断规则。

## 一、图书馆建法

对齐正规图书馆模式：**实体资源在云盘，Obsidian 里只存索引/卡片，不搬文件进来**。核心是**浏览 / 检索 / 策展 三样各司其职**（别硬塞进一个文件，也别过度建库）：

> **🔑 生成器铁律**：`00_馆藏总览` 和 `_全文检索/` 都由本 skill 的 `scripts/gen_catalog.py` 产出，**靠调用技能生成，绝不把脚本写进用户的 OB vault**（OB 只放产出）。扫描数据（JSONL）是运行时数据，放 `/tmp` 或私有 env，不进 vault 也不进 skill 仓库。
> **⚠️ 网盘深链格式（实盘校准，别猜）**：备份盘文件夹 = `https://www.alipan.com/drive/file/all/backup/<file_id>`（`folder/<id>` 会弹回首页）；用 `--url-prefix` 传，默认已是此格式；file_id 来自扫描数据。
> 命令：`python3 <本skill>/scripts/gen_catalog.py --scan-dir <扫描JSONL目录> --out <总览.md> --search-dir <检索目录> [--moves f --deletes f --roots f --junk <碎片前缀>]`

### 1. 00_馆藏总览（浏览：折叠树 + 资源级表格）
`gen_catalog.py` 从全盘扫描 JSONL 生成：**分类夹（`NN_` 前缀）→ 可折叠标题**（`#`/`##`/…，Obsidian 大纲面板直接跳转）；**第一个非 `NN_` 目录 = 一个"资源" → 表格一行**（资源名 🔗直达 / 媒介 / 文件数 / 大小）。点 🔗 到网盘看文件，**总览本身不铺 15 万个文件、保持清爽**（实测全铺=41k 行乱成一团，压到 ~600 行）。方括号 `[]`、竖线 `|` 等要转义避免破坏链接/表格。

### 1b. 全文检索索引（检索：搜到任意单文件）
`--search-dir` 触发：按区各出一份 `_全文检索/<区>.md`，每个资源一个 `##` 标题下**「文件 | 大小」表格**列出全部文件（相对路径）。**只搜不看**（Ctrl+F / 全局搜），文件大所以别在编辑模式久留；`--junk` 排除无检索价值的碎片区（如 Office 模板 8 万碎片）。
**❌ 反面教训（别重犯）**：不要为了做"全盘可筛选数据库"给每个文件夹建 frontmatter 桩笔记——Bases 只能查独立笔记，几千个机器桩、与总览重复、把 vault 撑翻倍，得不偿失（试过 4459 个后删光）。属性筛选只留给下面的精选卡；全文件检索用 `_全文检索/` 平铺清单即可。

### 2. 馆藏卡（每份精选资源一张，frontmatter 结构化）
一卡对应云盘一个资源目录，存于 `10_馆藏卡/<学科>/` 下，frontmatter 字段：

```yaml
---
type: 馆藏
tags: [馆藏]           # 必填，Bases 靠这个 tag 筛选，漏打就不出现在任何视图
medium: 视频课          # 视频课 / 纪录片 / 教辅 / 绘本 / ……
audience: 学习者A        # 适用对象，多个用顿号分隔；开源模板禁止写可识别个人信息
subject: 数学           # 学科
level: 小学高           # 学段
episodes: 186           # 集数/册数（真实值，来自 ls 实测，不是宣传页数字）
drive: 备份盘           # 所在云盘盘位
drive_link: https://www.alipan.com/drive/folder/<id>
map: "[[云盘地图-xxx]]" # 关联的板块级地图双链
status: 在架            # 在架 / 下架 / 待补充
---

# <资源名>

<一两句话说清楚这是什么、覆盖范围（如"38集全部平铺无子目录，覆盖XX/XX/XX典型题型"）>

**用法建议**：<针对用户本次提供的学习者代号/弱项写建议，不写可识别个人信息或长期个人档案>
```

`episodes` 一定要实测（云盘 `ls` 数出真实集数），不要照抄课程宣传页——本次整理中「用动画片学奥数」纠正为实测 38 集（原以为 20 集），排期方法论继承 reading-plan 的"按真实体量排期，不拍脑袋"。

### 3. Bases 数据库（`.base` 文件，多视图查询入口）
一个 `.base` 文件挂在馆藏目录根，`filter: file.hasTag("馆藏")`，至少建这几个视图：
- **全部馆藏**（table，按学科排序）
- **按学科**（table，groupBy subject）——回答"这个学科有什么资源"
- **按对象**（table，groupBy audience）——回答"哪个学习者/受众能用什么"
- **按媒介**（table，groupBy medium）
- **卡片墙**（cards，groupBy subject）——视觉浏览

`properties.displayName` 把英文字段名映射成中文列名（资源名/学科/适用/媒介/集数/学段/状态），不需要用户认字段名。

### 4. 分区深度分类方案文档
每个一级/二级分区一份 `<分区名>-深度分类方案.md`，frontmatter 带 `status`（`✅ 已执行` / `待拍板`），内容结构：
- 现状问题描述（一两句，量化：几套/几目录平铺）
- N 大类结构（代码块画树）+ 归类映射表（原编号→归入哪类）
- 归类规则（供**以后新增资源**落位判断，不只是这一次的移动清单）
- 变更史（时间戳 + 已完成动作）
- 待办/边界模糊项（如两个分区交界处如何判断，预防性写规则，不强求马上处理现有条目）

## 二、分类方法论

### 2.1 受众三分（学习者直接用 / 照护者或教师学习 / 机构教研）
教育类资源库最容易和"学习者直接用的东西"混在一起的两类是：
- **照护者/教师向**：教育方法课、教养书籍、护理教程——成人自己学的，不是给学习者直接看的
- **机构教研向**：教案、优质课汇编、机构运营资料——面向幼师/机构从业者，普通家庭场景通常用不上

**判断流程（审计表逐项标注再迁移，不凭印象一次性搬）**：
1. **关键词初标**：按名称关键词粗筛（含"育儿/父母/家长/家教/护理/育婴/教案/园/机构/优质课"等词 → 初步标家长或园所，其余初步标孩子）
2. **人工校正误标**：关键词初标必有漏网之鱼（例如某些早教计划字面像学习者直接用，实为成人操作手册；某些教案内混着员工手册应归机构），逐条过一遍人工修正
3. **落审计表**：产出一张表——`# | 二级夹 | 套名 | 归属`，覆盖该分区**全部条目**，不是只列有疑问的
4. **归属模糊单独列待裁定区**：判断不了的（如"亲子综合班"到底是家长能直接操作还是需要专业机构话术）单独列一节「待用户裁定项」，写清疑点和建议裁定方向，不要为了推进而随手拍板
5. **裁定后再迁移**：方案先出（status: 待拍板），用户对疑难项拍板后，才生成物理移动清单——**不要方案和执行揉在一起一次性做完**

成人向内部可再分（教育方法课 / 教育书籍 / 护理资料），机构向去向给用户二选一（迁到成人阅读存档区 vs 留在原库改名标注），不要替用户预设答案。

### 2.2 学段序号化
一二级目录全部加 `NN_` 数字前缀，**数字即浏览顺序**（不是创建时间顺序）：
- `10/20/30…90` 步进，留出中间号方便以后插入新分区不用全部重编号
- `90_` 固定语义是"存档/待拆类"，全库统一，看到 90 就知道是收尾区
- 学段类目按认知发展顺序编号（如启蒙→绘本→小学→中学→素养课→影视），不是按字母或添加时间

### 2.3 杂包必拆
遇到"XX资源""XX合集"这种名字含糊、内部混装多种性质资源的杂包目录：
1. **逐项盘点**：先把包内每一项都列出来（`ls` 一层），不要看包名就假设内容
2. **宝贝提升为学科区一级条目**：包内如果混着体量大/质量高的独立资源（本次案例：混装包里的牛津阅读树 50G、小灯塔通识合集、芝麻学社 95 门），这类"宝贝"要**提升为对应学科区的一级条目**，不能让它埋没在杂包名字里
3. **剩余分类**：包内剩下的按各自性质分到该分的类
4. **删壳对账**：包内容全部搬空后，**对账总数**（原包内 N 项 = 提升 M 项 + 归类 K 项，M+K=N 才能确认没漏），对上后删除空壳目录

本次「小学生学习资源」杂包 17 项对账吻合后拆净，即遵循此流程。

### 2.4 SHA1 级查重
遇到"疑似重复"（同名资源出现在两个位置、或版本号/来源不同但内容像是一样的），**先做文件级 SHA1 哈希比对，再决定删哪份**——不能只凭文件名/大小相似就删，万一是不同版本/不同清晰度会误删。

哈希一致 → 放心删掉多余那份；哈希不一致（如画质不同、集数不同）→ 保留更优版本，在分类方案里记一笔说明为什么留这份删那份。本次「窦神大语文」查重去重、「黑客帝国三部曲套娃重复」都是 SHA1 比对确认一致后才删除的。

### 2.5 广告清理特征清单
资源包里混的营销/导流内容有固定套路，见到以下特征直接判定为广告，批量清理：
- 目录名/文件名带「几千T资源」「XX00T网盘地址」类夸大体量宣传
- 「扫码进群」「↑↑↑点左上角头像订阅↑↑↑」类导流图片/文字
- 「十万度V信」「XH1080.com」类论坛/站点尾巴（通常加在分集文件名末尾）
- 「超低价阿里云盘会员」类广告目录
- 打包内藏的可疑 `.exe`（本次案例：「打包防和谐.exe」类，与课程内容无关的可执行文件，从来源不明的资源包里发现即应删除，不要运行）

清理原则：**文字类广告见到即删**（低风险，不需要用户确认）；**可疑 exe 类先列清单让用户确认**再删（不排除有极小概率是课程配套工具，虽然本次案例都判定为纯广告）。

## 三、组织流程（四步闭环）

1. **方案先行**：分类方案文档先写完整（现状/N类结构/归类规则/待裁定区），frontmatter `status: 待拍板`，不擅自开始移动
2. **用户裁定**：疑难项（受众模糊/去向二选一/查重后留哪份）列清单给用户，用户对每项拍板
3. **分批执行**：裁定后按"风险从低到高、跨库/跨盘操作放在后面"分批（如：批次1同库内重分层 → 批次2跨库迁出 → 批次3待裁定后追加），**每批做完立即复核**（重新 ls 对照终态），操作过程记一份移动日志（jsonl，字段含：原路径/新路径/操作类型/时间戳），不要等全部批次做完才一起验证
4. **地图/总览刷新 + 文档回填**：结构变了之后，受影响的地图文件重新扫描重建（尤其跨盘移动导致 file_id 全换的情况），00_馆藏总览同步更新链接，分类方案文档的 `status` 改成 `✅ 已执行` 并补一段"变更史"记录本次做了什么

## 四、执行编排经验

- **决策与执行分层**：主控角色（用户本人/Fable5 类主脑）只做**审核决策**——批准分类方案、拍板疑难项、复核终态；实际的 `ls`/`mv`/`rename` 操作交给执行角色（sonnet 级代理）去跑。不要让决策者亲自敲每一条命令，也不要让执行者自己决定分类逻辑。
- **并行必须给进度看板**：多个执行代理同时跑不同分区时，主控方需要一份可扫一眼看完的进度看板（哪个分区在跑/完成/待复核），不要让用户去追多个并发会话各自的输出。
- **写操作双层验证**：任何 `mv`/`rm`/`rename` 类写操作，执行完成后立即用**独立的验证通道**（如换一次 `ls` 请求、或用 Read 工具而非 Bash 工具重新确认）核对结果，不要相信"命令返回了成功提示"就等于真的执行了——本次整理战役中出现过至少一次"回显污染"（工具输出显示成功但实际未执行），靠双层验证发现并纠正。
