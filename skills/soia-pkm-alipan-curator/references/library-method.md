# 图书馆建法与整理方法论（library-method）

> 来源：多分区阿里云盘整理经验。补充 SKILL.md 未展开的"怎么把云盘变成图书馆"和"怎么定分类方案"，只保留可复用方法，不包含个人目录或历史资源数据。

## 一、图书馆建法

对齐正规图书馆模式：**实体资源在云盘，Obsidian 里只存索引/卡片，不搬文件进来**。核心是**浏览 / 检索 / 策展 三样各司其职**（别硬塞进一个文件，也别过度建库）：

> **🔑 生成器铁律**：馆藏总览和全文检索都由本 skill 的 `scripts/gen_catalog.py` 产出，**绝不把脚本写进用户 vault**。扫描 JSONL 放 `${TMPDIR:-/tmp}/soia-pkm-alipan-curator/` 或用户显式指定的运行目录，不进 vault/仓库。
> **⚠️ 网盘深链格式**：不同盘位可能不同，调用方必须传 `--url-prefix <drive-url-prefix>` 或设置 `SOIA_ALIPAN_URL_PREFIX`；file_id 来自扫描数据，公共 skill 不写死盘位。
> 命令：`python3 <skill-dir>/scripts/gen_catalog.py --scan-dir <scan-dir> --out <catalog-output> --search-dir <search-output-dir> --url-prefix <drive-url-prefix> [--moves f --deletes f --roots f --junk <ignored-prefixes> --heading-pattern REGEX --section-icons JSON --max-heading-depth N]`

### 1. 馆藏总览（浏览：业务目录标题 + 素材文件夹表格）
`gen_catalog.py` 从全盘扫描 JSONL 生成。默认保守展示全部目录，保证无编号体系不会被漏掉；需要折叠内部素材树时，通过 `--heading-pattern` 提供业务目录正则，例如编号体系使用 `^\d{2}[_.]`。标题显示当前实体目录名并可直达；不匹配的内部素材目录在最近业务标题下聚合。根分区默认使用中性 `📁`，需要语义图标时通过 `--section-icons '{"关键词":"图标"}'` 注入，生成器不写死用户分类名称。标题最多到 H6；总览本身不铺单个文件。

### 1b. 全文检索索引（检索：搜到任意单文件）
`--search-dir` 触发：按区各出一份检索 Markdown，每个资源一个 `##` 标题下用**「文件 | 大小」表格**列出文件（相对路径）。**只搜不看**；`--junk` 排除用户明确指定的无检索价值碎片区。
**❌ 反面教训（别重犯）**：不要为了做"全盘可筛选数据库"给每个文件夹建 frontmatter 桩笔记——大量机器桩会与总览重复并显著放大 vault。属性筛选只留给精选卡；全文件检索使用生成器输出的分区检索文件。

### 2. 馆藏卡（每份精选资源一张，frontmatter 结构化）
一卡对应云盘一个资源目录，存于用户指定的 `<vault>/<card-relative-dir>/[<可选分组>/]`。最低字段只有 `type`、`tags` 和 `drive_link`；下列其他字段均按本次主分类轴和查询需求选用：

```yaml
---
type: 馆藏
tags: [馆藏]           # 必填，Bases 靠这个 tag 筛选，漏打就不出现在任何视图
medium: 视频课          # 可选：只有媒介是常用筛选维度时才填
audience: <适用受众>     # 可选；多个用顿号分隔，不写可识别个人信息
topic: <主题或领域>      # 可选：仅当主题/领域是本次轴或常用筛选维度
stage: <阶段或难度>      # 可选；仅在本领域有稳定阶段时填写
item_count: <实测数量>   # 文件、集数、册数等实际资源单位
drive: <盘位显示名>
drive_link: <用户盘位的目录直达链接>
map: "[[<map-wikilink-target>]]"
status: 在架            # 可选：仅在需要生命周期管理时填写
---

# <资源名>

<一两句话说清楚这是什么、覆盖范围和实测体量>

**用法建议**：<根据本次目标写建议，不写可识别个人信息或长期个人档案>
```

`item_count` 一定要实测，不要照抄宣传页；视频系列可额外记录 `episodes` 和单集时长，文档合集可记录册数/页数。学习计划按真实体量排期。

### 3. Bases 数据库（`.base` 文件，多视图查询入口）
一个 `.base` 文件挂在馆藏目录根，`filter: file.hasTag("馆藏")`。保留“全部馆藏”作为稳定入口；其余视图根据用户确认的主分类轴和常用问题生成，例如按主题、按受众、按媒介或按状态。没有对应字段时不建空视图。

`properties.displayName` 把英文字段名映射成用户可读列名。字段集合跟随本次分类方案，不强制所有用户都有学科、学段或集数。

### 4. 分区深度分类方案文档
每个一级/二级分区一份 `<分区名>-深度分类方案.md`，frontmatter 带 `status`（`✅ 已执行` / `待拍板`），内容结构：
- 现状问题描述（一两句，量化：几套/几目录平铺）
- N 大类结构（代码块画树）+ 归类映射表（原编号→归入哪类）
- 归类规则（供**以后新增资源**落位判断，不只是这一次的移动清单）
- 变更史（时间戳 + 已完成动作）
- 待办/边界模糊项（如两个分区交界处如何判断，预防性写规则，不强求马上处理现有条目）

## 二、分类方法论

完整方法见 **[classification-methods.md](classification-methods.md)**。先真读代表内容，再从主题、用途、受众、阶段、状态、媒介等维度选择本层主轴；教育、编程、读书等结构都只是领域示例，不能直接复制成所有用户的目录。

图书馆层只补充四条约束：

1. 分类方案必须覆盖全部资源根，并为每项记录内容证据、置信度和建议去向；不能只列疑问项。
2. 目录用于稳定浏览，媒介、受众、状态等次要维度优先进入馆藏卡或 Excel 筛选列，避免同层混轴。
3. 编号是用户可选的排序方式；格式、步长和待核/存档编号由本次方案声明，公共 skill 不设固定值。
4. 杂包逐项盘点，独立资源整包移动；课程或技术包内部的讲义、代码、字幕、素材不按媒介拆散。
5. 若同时存在课程区和书库，先写互斥分区边界：独立出版物/独立书集可归书库，课程组成材料随课程整包保留；具体边界由用户确认，不把该示例写成公共固定目录。
6. 用户选择编号、学习导览或待确认区时，把适用父目录、格式/名称和复核根写入结构合同；终态用 `scripts/audit_structure.py` 验证，不能仅凭索引外观宣布完成。

疑似重复仍须做文件级 SHA1 比对；广告与可疑可执行文件仍须先列候选清单并取得授权。删除与去重证据见 `operations-troubleshooting.md`。

## 三、组织流程（四步闭环）

1. **方案先行**：分类方案文档先写完整（现状/N类结构/归类规则/待裁定区），frontmatter `status: 待拍板`，不擅自开始移动
2. **用户裁定**：疑难项（受众模糊/去向二选一/查重后留哪份）列清单给用户，用户对每项拍板
3. **分批执行**：裁定后按"风险从低到高、跨库/跨盘操作放在后面"分批（如：批次1同库内重分层 → 批次2跨库迁出 → 批次3待裁定后追加），**每批做完立即复核**（重新 ls 对照终态），操作过程记一份移动日志（jsonl，字段含：原路径/新路径/操作类型/时间戳），不要等全部批次做完才一起验证
4. **地图/总览刷新 + 文档回填**：结构变了之后，受影响的地图文件重新扫描重建（尤其跨盘移动导致 file_id 全换的情况），00_馆藏总览同步更新链接，分类方案文档的 `status` 改成 `✅ 已执行` 并补一段"变更史"记录本次做了什么

## 四、执行编排经验

- **决策与执行分层**：主控角色只做**审核决策**——批准分类方案、拍板疑难项、复核终态；执行角色只在已批准范围内运行 `ls`/`mv`/`rename`，不自行决定分类逻辑。
- **并行必须给进度看板**：多个执行代理同时跑不同分区时，主控方需要一份可扫一眼看完的进度看板（哪个分区在跑/完成/待复核），不要让用户去追多个并发会话各自的输出。
- **写操作双层验证**：任何 `mv`/`rm`/`rename` 类写操作，执行完成后立即用**独立的验证通道**重新确认，不把命令回显当成终态证据。
