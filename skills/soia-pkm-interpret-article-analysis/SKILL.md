# soia-pkm-interpret-article-analysis

> 属 SOIA 个人知识管理域（`soia-pkm-*`）的"解读"环节：对 vault 里 clip 进来的长文/论文，AI 直接给出结构化解读，帮你在深挖（`distill`）前先判断这篇值不值得投入。

## 客户可读说明

### 这个技能可以做什么

对 vault 里 clip 进来的长文/论文，AI 直接给出解读：内容总览/核心要点/关键启发/批判视角/延伸阅读五段式。默认快读（各段 2-3 句），说"精读/深度解读"升级为逐节展开 + 论证链核查。产出独立 `<原文件名>-AI解读.md`，落原文件同目录并双链回原文，绝不碰原文、绝不写入原文的 `## 我的看法` 段。

| 客户想要 | 技能会做 | 客户能看到 |
|---|---|---|
| 快速判断一篇长文/论文值不值得细读 | 快读：五段式各 2-3 句的 AI 解读 | `<原文件名>-AI解读.md`，双链回原文 |
| 精读一篇重要文章，要逐节核查论证 | 精读：逐节展开 + 论证链核查，标出原文未明确处 | 更详细的解读文件，批判视角至少一条 |
| 批量解读同一批文章（超过 3 篇） | 先报清单（文件名 + 主题），等待确认范围 | 待解读清单 → 逐篇解读文件 + 汇总回执 |
| 想把这篇炼成自己的观点，而不是看 AI 怎么想 | 提示这不是本技能职责，指向 `soia-pkm-distill-article-opinion` | 边界说明，不会被误当成你的观点 |

### 客户如何使用

1. 提供要解读的文章：说"这篇"（=当前对话涉及的文章）或给出文件名/标题，Agent 在 `<vault-articles-dir>/` 下定位。
2. 单篇文章：Agent 直接执行，不需要额外确认。批量（同一请求涉及超过 3 篇）：Agent 先列出清单（文件名 + 大致主题），等待客户确认解读范围（全部 / 挑选几篇 / 暂停），再逐篇执行。
3. 深度默认**快读**（五段式，各段 2-3 句）；客户说"精读""深度解读""逐节核查"时升级为**精读**（逐节展开 + 论证链核查，见 [两档深度](#两档深度)）。
4. 解读永远**先落盘、后汇报**：产出 `<原文件名>-AI解读.md` 落在原文件同目录，正文双链回原文，绝不改动或覆盖原文件（含 `## 我的看法` 段）。
5. 最终回复必须给客户完整回执：深度、篇数、产出文件路径、标注了几处"原文未明确/需核对"、剩余风险。

### 依赖与安装

安装本技能（单个技能）：

```bash
npx skills add soia-team/soia-open-skills -g -a '*' -s soia-pkm-interpret-article-analysis -y
```

| 依赖 | 类型 | 安装 / 配置 | 缺失时怎么处理 |
|---|---|---|---|
| Obsidian vault（任意结构） | 方法参考，非强依赖 | 不需要单独安装 | 本技能可以解读任意 Markdown/纯文本文章，不要求文件必须在 vault 内；只有涉及 vault frontmatter 和双链归位约定时才参考 vault 上下文 |
| `soia-pkm-clip-*` 系列 | 上游来源，非强依赖 | 不需要安装，除非客户需要先归档再解读 | 只是常见的输入来源，本技能不调用也不修改这些 skill 的任何文件 |
| `soia-pkm-distill-article-opinion` | 下游/平行 skill，非依赖 | 不需要安装 | 本技能不写入 `## 我的看法`；客户要"我的观点"时提示改用它，不代替执行 |

配置约定：

```text
~/.config/soia-skills/soia-open-skills/soia-pkm/soia-pkm-interpret-article-analysis/config.yml
SOIA_PKM_INTERPRET_ARTICLE_ANALYSIS_CONFIG_FILE=<custom-config-path>
```

- 本技能是纯文档 / 纯 prompt 技能，不带脚本，通常不需要 `config.yml`；如果客户想固定"默认深度"或"批量确认阈值"等偏好，可以建这个文件，字段与本文一致。
- 不需要、也不应该读取任何 API key、cookie、session、账号凭据——本技能只读取客户提供的文本文件。

## 与 soia-pkm-distill-article-opinion 的分界（必须遵守）

**这是本技能最容易被混淆的地方——分界必须显著、每次执行前默认成立：**

| 维度 | soia-pkm-interpret-article-analysis（本技能） | soia-pkm-distill-article-opinion |
|---|---|---|
| 解读主体 | **AI** 通读全文给出判断 | **用户**口述，AI 只帮问、帮落文字 |
| 交互方式 | 一次性生成五段式解读，不逐问逐答 | 苏格拉底式，一次只抛一个问题，等用户回答 |
| 产出内容 | AI 的总览 / 要点 / 启发 / 批判 / 延伸 | 用户自己的立场、经验、判断 |
| 落地位置 | 独立文件 `<原文件名>-AI解读.md` | 原文件内的 `## 我的看法` 段 |
| 使用时机 | 刚归档、还没读完，想先判断值不值得深挖 | 已经决定要对这篇表态、要炼观点 |
| 红线 | **绝不写入原文 `## 我的看法` 段**——那是用户专属 | **绝不替用户表态**——内容必须来自用户回答 |

- 客户如果在解读过程中说"我怎么看这篇""帮我提炼观点""这篇我的看法是……"，一律提示"这是 `soia-pkm-distill-article-opinion` 的职责，要不要切过去？"，不在本技能内代答、代写。
- interpret 的产物是**决策辅助**：帮客户判断"这篇值不值得进 distill 深挖"；不是观点本身，也不应被误当成客户的观点。

## 两档深度

| 深度 | 触发词 | 输出形态 |
|---|---|---|
| **快读**（默认） | 未明确要求精读时 | 五段式，每段 2-3 句，覆盖全文但不逐节展开 |
| **精读** | 「精读」「深度解读」「逐节核查」 | 五段式基础上，逐节展开 + 论证链核查（见下） |

### 精读模式：论证链核查

精读时，在"批判视角"段之前先做一次逐节论证链核查，逐个论点确认：

1. 这一节的核心论点是什么？
2. 支撑这个论点的证据/数据/引用是什么，出自原文哪一段？
3. 论点到证据之间是否有跳跃——证据只能支持一个更弱的结论，但原文喊出了更强的结论？
4. 原文没有交代清楚的前提/定义/边界条件，标注"原文未明确/需核对"，不替原文补全。

核查结果作为"批判视角"段的依据，而不是另起一段罗列；避免精读产出变成"总览 + 核查清单"两张皮。

## 五段式解读结构（输出模板）

产出文件 `<原文件名>-AI解读.md` 正文结构固定为：

```markdown
# AI 解读：《{文章标题}》

> 原文：[[<原文件名>]]
> 解读深度：{快读 / 精读}
> 解读时间：{YYYY-MM-DD}

---

## 内容总览

{快读 2-3 句 / 精读可展开到一小段：文章讲了什么、核心结论是什么}

## 核心要点

{分条目提炼文章主干信息，通常 4-8 条，每条给出处（如"见第 3 节"/"见引言部分"）}

- **要点 1**（出处：……）：……
- **要点 2**（出处：……）：……

## 关键启发

{至少 3 条，每条说明"是什么 → 为什么重要 → 对你可能怎么用"，不写"这个方法很有前景"式空话}

- **启发 1**：……
- **启发 2**：……
- **启发 3**：……

## 批判视角

{至少 1 条"这篇可能错在哪"——论证漏洞、证据不足、以偏概全、时效性问题等；精读模式基于上面的论证链核查填写；不确定处写"原文未明确/需核对"，不替作者圆场也不无端苛责}

- ……

## 延伸阅读

{2-4 条值得继续探索的方向或值得对照读的资料；如果 vault 内已有相关文章/MOC，用双链标出}

- ……
```

- 不确定内容一律显式标"原文未明确/需核对"，绝不编造原文没写的数据、实验、结论。
- 引用原文时给出处段落（章节标题、"引言"/"第 X 段"等），方便客户回原文核对。

## 定位文章

1. 客户说"这篇" = 当前对话涉及的文章；否则用文件名/标题在 `<vault-articles-dir>/` 下 Glob/Grep 定位。
2. 读文章的 `## 原文` 段（clip 家族统一正文结构：`## 摘要 / 原文 / 我的看法 / 关联`）；只有摘要没有全文时，解读前先告知客户"本次基于摘要而非全文，结论置信度较低"，并在解读文件的"内容总览"里注明。
3. 不读、不改客户已有的 `## 我的看法` 段——那段内容与本技能无关。

## 落地规则

- 路径：与原文件**同目录**，文件名 `<原文件名>-AI解读.md`。
- frontmatter：新建独立 frontmatter（不是原文件 frontmatter 的复制），至少包含：

```yaml
interpreted_from: <原文件名或相对路径>
interpret_depth: <quick 或 deep>
interpreted_at: <YYYY-MM-DD>
```

- 正文用 `[[<原文件名>]]` 双链回原文，方便双向跳转。
- **绝不覆盖、绝不编辑原文件**——包括 frontmatter、`## 摘要`、`## 原文`、`## 我的看法`、`## 关联`任何一段；如果同名解读文件已存在，先告知客户，等待确认是否覆盖旧解读（覆盖类风险动作，必须先确认）。

## 批量策略

- 单篇（本次请求只涉及 1 篇）：直接执行，不额外确认。
- 批量（本次请求涉及超过 3 篇，如"把这个 MOC 下的文章都解读一下"）：先列出清单（文件名 + 一句话主题猜测），报告"共 N 篇，建议先解读哪几篇/全部解读？"，等待客户确认范围，再逐篇执行；每篇仍遵守"先落盘、后汇报"，全部完成后给一份合并回执（见 [日志与完成回执](#日志与完成回执)）。
- 2-3 篇：视为小批量，可以直接执行，但仍在开始前用一句话报告将处理哪几篇。

## 交付顺序

先落盘，后汇报：产出文件必须先写入磁盘并通过内容核对（文件确实存在、五段式齐全、批判视角至少一条），再向客户发出完成回执。不允许先描述"已完成"再补写文件。

## 边界与限制

- **不代替 `soia-pkm-distill-article-opinion`**：本技能绝不写入原文件的 `## 我的看法` 段，也不以第一人称冒充客户表态。
- **不编造**：原文未提供的信息一律标"原文未明确/需核对"，不杜撰数据、实验、引用或结论。
- **不做外部检索核实**：解读基于原文本身；如果批判视角依赖外部事实核查，只提示客户"这一点需要客户自行核实"，不擅自杜撰核实结果。
- **不做翻译/转换**：外文文章先经 `soia-pkm-translate-article-zh` 或客户自行理解后再解读；本技能不覆盖 `soia-pkm-transform` 的产物转换（PPT/脑图等）职责。
- 本技能是纯文档 / 纯 prompt 技能，不带任何脚本，不读取账号、cookie、session 等凭据。

## 闭环位置

```
clip(收) → organize(整理) → ★interpret(AI 解读，辅助判断) → distill(点：你的观点) → compose(写) → publish(发)
```

- 上游：`soia-pkm-organize-article-moc` 整理好的文章库是主要输入来源；也可以直接对刚 `clip` 进来、还没整理的单篇文章解读。
- 下游：客户看完 AI 解读、决定"这篇值得深挖"后，转 `soia-pkm-distill-article-opinion` 炼自己的观点；`soia-pkm-interpret-article-analysis` 的产出不是 `distill` 或 `compose` 的必需输入，只是决策参考。

## 第三方 skill 关系

| 第三方 skill | 对 `soia-pkm-interpret-article-analysis` 的关系 |
|---|---|
| `soia-pkm-distill-article-opinion` | 分工边界见上表；本 skill 不调用它，也不写它专属的 `## 我的看法` 段 |
| `soia-pkm-clip-*` 系列 | 常见上游来源，非依赖；本 skill 不修改这些 skill 的文件 |
| `soia-pkm-organize-article-moc` | 常见上游，非依赖；本 skill 不要求文章已被 organize 整理过 |
| `soia-pkm-translate-article-zh` | 外文文章的翻译前置，非依赖；本 skill 不做翻译 |

## 完成后回执

执行完**必须**向用户输出（不要默默做完）：

1. **做了什么** — 一句话总结本次解读了什么文件、什么深度、是否批量。
2. **文件变化** — 新建的解读文件完整路径；明确说明原文件（含 `## 我的看法`）未被改动。
3. **下一步** — 提示客户"这篇值不值得深挖"的判断依据；快读模式提醒可回复"精读"升级；有"原文未明确/需核对"的点时一并列出，建议客户自行核实或用 `soia-pkm-distill-article-opinion` 表态。
