# 模式 D 规格编译参考

本文件承载模式 D 的详细规则，按需加载。正文保留模式选择入口和不可违反的质量底线；进入模式 D 前先阅读本文件，并按其中的需求覆盖账本、事实来源、压缩和回执要求执行。

## 内容导航

- 统一调用契约、输入输出字段与未知值处理
- 预检、模型完整性、断点恢复与完成回执
- 危险目录与反虚假修复门禁

## 模式 D · 扩展成规格（需求编译器，"放"模式）

> **进入模式 D 前必读**：`references/mode-d-quality-gate.md`——模型无关质量 Rubric、硬失败条件、≥10 个回归样本（含复杂工程需求基线案例）与前向验证协议。模式 D 交付前必须对照该文件自检，不是可选阅读；静态自检通过不等于目标模型已经完成前向验证。

B/C 偏"收"（诊断、缩减、消歧），本模式是"放"：把模糊、自相矛盾、或"要求完备却没给验证方法"的需求，**编译成可机械验证的工程规格**，而不是把模糊的部分删掉或套一份通用模板。典型输入："测试所有模型所有深度，必须无遗漏"（自相矛盾：范围开放却要求完备）。典型输出：需求覆盖账本 + 完整目标矩阵 + 状态枚举 + 断点恢复 + 逐条验收标准 + **可直接复制执行的完整提示词**。

### 先判断要不要进入模式 D（规格密度控制·入口闸）

**简单任务不进入模式 D，继续走 A/B/C。** 只有任务具有明显工程复杂度或高风险，且需求命中以下至少一项，才优先考虑 D：多对象覆盖、多阶段流程、要求"全量/不得遗漏"、需要状态管理、涉及成本治理、需要失败恢复、或有严格验收要求。单独出现"成本"、"验收"等词不构成模式 D 的充分条件；反例见 `references/mode-d-quality-gate.md` 的回归样本——一句润色请求、一次简单费用计算，或"提示词冗长但需求本身明确"的优化请求，都不应该被拉进矩阵、状态机或 schema。

### 阻断性判定闸（先判再问）

模式 D 只有在缺失事实会改变以下任一项时才暂停产出并一次问全：任务意图、目标受众 AI、最终输出形态、所有权/授权红线、不可逆动作范围，或必须由客户决定的公共契约。用户已经要求执行者动态盘点、提供了占位符/资料路径，或把结构设计委托给 Agent 时，相关细节均为**非阻断项**。

以下内容默认不构成阻断，除非客户明确要求先由自己决定：未来执行者可从文件/CLI/仓库探测的数据；尚未给值的显式占位符；YAML 文件名与字段组织；矩阵枚举方式；状态名；断点文件格式；路由权重与默认值。把它们写成 `<待执行者读取>`、`unknown/待探测`、候选结构或执行阶段决策，不要先追问客户。

### 源码真身与安装入口（涉及 skill 开发时）

`~/.agents/skills`、`~/.codex/skills`、`~/.claude/skills`、`~/.gemini/antigravity-cli/skills`、`~/.workbuddy/skills` 等默认是安装或消费入口，不是开发源仓库。`.skill-lock.json` 只能证明来源仓库与 `skillPath`，不能单独证明本机 checkout 的位置。

生成修改、测试或发布 skill 的提示词时：

1. 客户已给 source repo path：直接使用该路径。
2. 客户未给 source repo path：使用 `<source-repo-path>` 占位，并要求执行者定位带 `.git` 的 checkout、用 `git remote get-url origin` 核对来源，再拼接已记录的 `skillPath`。
3. 不得把安装目录、软链接目标或 lock 记录直接断言为源码真身。
4. 先读取源码仓库中已有的 `SKILL.md`、scripts、references 与配置，优先复用和扩展现有实现；只有客户已提供或执行者已发现的文件，才能写成确定存在的具体路径。
5. 若要求远程安装验收，顺序必须是：修改源码仓库 -> 仓库校验 -> 获得授权后 commit/push -> 从远程执行 `npx` 重装 -> 对比安装产物。local install 只能用于调试，不能冒充最终验收。

客户明确要求"给出完整提示词"时，只要需求覆盖账本中没有真正的 `clarification_required` 项，就必须先交付完整提示词；非阻断问题最多放在产出后的"可选确认项"，不得用一份澄清清单代替交付。结构选择属于本模式可以创造的部分，不能反过来要求客户替 Agent 完成规格设计。

### 第一律：结构可以造，事实不能造

- **可以自由创造的**：矩阵组织方式、状态枚举、字段 schema、验收标准、停止条件、恢复机制——这些是"结构"，是本模式的价值所在。
- **绝不允许虚构的**：模型 ID、价格数字、CLI 参数、现有能力、历史事故——这些是"事实"，每一条都必须来自原料或标注 unknown/待探测。
- **不硬编码进本技能的具体领域事实**（这些事实属于目标领域 skill、用户输入或对应 reference，本技能只负责提醒执行者去加载/探测，不替它们背书具体数值）：具体 Codex/Claude 等 model ID、模型实时价格、目标技能的完整 schema、某个 CLI 的固定参数、某个项目的目录结构、某次事故的不可复用细节。

### 需求编译流程（Capture → Deliver，八阶段）

| # | 阶段 | 做什么 |
|---|------|--------|
| 1 | **Capture** | 提取用户全部显式要求，逐条列出，不省略、不合并 |
| 2 | **Classify** | 把每条要求归类为事实 / 目标 / 约束 / 偏好 / 风险 / 未知项 |
| 3 | **Ground** | 给事实项标来源（原料引用或探测命令）；未知事实标"待探测" |
| 4 | **Expand** | 把模糊要求转成矩阵/状态/字段/流程/验收标准（见下方「正向转换规则」） |
| 5 | **Trace** | 建立"需求 → 提示词条款"的覆盖关系，写入需求覆盖账本 |
| 6 | **Challenge** | 检查需求冲突、隐含失败路径、最脆弱假设——攻击自己编译出的结构 |
| 7 | **Compress** | 删除不改变行为的冗余结构，产出最小充分规格（见下方「规格密度控制」） |
| 8 | **Deliver** | 输出可直接复制执行的完整提示词 + 简短说明；不强迫用户看到 1–7 的完整内部推理，但最终回执必须证明需求没有丢失 |

高级模型可以合并、重排或用更优的等价流程替代以上八阶段，只要最终仍能证明"需求没有丢失"，且遵守第一律。

### 原料状态与"原料先行"的边界

本技能在**仅产出提示词**阶段，只使用客户在对话中提供的文本和明确路径信息，不读取客户的实际账号、vault 或本机文件——这与全局边界一致（见「边界与限制」），模式 D 不例外。原料分三类：

| 原料状态 | 含义 | 处理方式 |
|---|---|---|
| **已提供内容** | 客户把文件全文/数据粘贴进对话 | 可直接作为事实来源，标注【原料:对话】 |
| **已提供路径但未读取** | 客户只给了文件路径、命令或 URL，本会话没有实际打开 | **不得**假装已知道其中内容；把路径原样写入生成的执行提示词，要求未来执行者读取，标注【待执行者读取:路径】 |
| **待补充事实** | 路径、内容都没有，且规格依赖这项事实 | 阻断性缺失→一次问全；非阻断性缺失→占位符 `<待确认>` 或 `unknown/待探测`，不删需求 |

只有客户在当前消息明确要求「写完就执行」时，当前会话才进入文件读取和执行阶段（见「交付方式」）；此前"有路径"不等于"有内容"，不得因为拿到了路径就把文件内容当已知事实来写规格，也不得因为缺一项次要事实就删掉整条需求。

### 需求覆盖账本（Requirement Coverage Ledger）

Requirement ID 必须按客户原始需求顺序稳定编号。即使正文重新分组，也只能通过本台账映射，不得复用、错标或重新解释 `需求 #N`；若正文标号会增加噪声，就只在台账中编号，不在正文重复标注。

模式 D 必须为用户的**每一条**显式需求建立账本条目，状态只能是以下七种之一，禁止"没有状态、直接消失"：

| 状态 | 含义 |
|---|---|
| `preserved` | 原意完整保留 |
| `expanded` | 已转化成更可验证的结构 |
| `clarification_required` | 缺阻断性事实，需先澄清 |
| `placeholder` | 非阻断性细节暂用占位符 |
| `deferred` | 经用户明确同意后延期 |
| `rejected` | 因真实边界无法纳入，已说明原因 |
| `conflict` | 与另一条需求冲突，需用户决定 |

**交付前自检清单**（对照不过关就返工，不能带着问题交付）：

1. 所有需求是否都进了账本、都有状态。
2. `must` 有没有被静默改成 `optional`。
3. 自动动作有没有被静默改成"只给建议"。
4. 全量范围有没有被静默缩成抽样。
5. 客户要求的文件/对象有没有被不合理排除。
6. 客户要求的验收有没有被删掉。
7. 新增的每一处结构是不是真的服务于账本里某一条需求（不是为了看起来完整）。

账本在最终回复里压缩成简短表格即可，不要求把 Capture~Challenge 的内部推理全部展示给用户。

### 模糊需求的正向转换规则（认知脚手架，非固定字段）

以下是把常见模糊表述转成可验证结构的示例映射，**用来提醒自己别漏想，不是要求每份规格都塞满全部十条**：

| 模糊表述 | 转换方向 |
|---|---|
| "全部、不得遗漏" | 先盘点范围 → 生成目标矩阵 → 定义终态和完成条件 |
| "最佳模型/最佳方案" | 明确决策因子、优先级、用户覆盖权、选择理由 |
| "避免降级" | requested value 与 actual value 一致性检测、显式状态、禁止静默替换 |
| "控制成本" | 计量来源、计算公式、未知值处理、预算预检、停止条件 |
| "失败后继续" | 状态持久化、断点、恢复入口、幂等规则 |
| "必须验证" | 具体证据、测试命令或可机械检查的验收条件 |
| "支持所有工具" | 明确宿主、执行器、adapter、能力探测、未实现状态 |
| "不要影响现有能力" | 回归清单和不变量 |
| "根据难度自动选择" | 难度维度、路由规则、用户显式指定优先级、无静默回退 |
| "信息不足" | 阻断项先问，非阻断项占位符并显式标记 |

### 规格密度控制（Compress 阶段落地规则）

模式 D 不是"越长越好"，目标是**最小充分规格**：

1. 每增加一个 schema / 矩阵 / 状态 / 章节，先问"删掉后 AI 的行为或验收会不会变化"——不会变的必须删。
2. 固定字段数量、固定章节数量、固定验收条数**不是**质量指标。
3. 能用表格就不要写成长段。
4. 提示词长度必须与任务风险和复杂度成比例，不是与"看起来专业"成比例。
5. 轻量模型按脚手架走全套流程防遗漏，但**仍必须执行 Compress 阶段**，不能跳过精简直接交付。
6. 高级模型可以用更短、更优的等价结构，不需要严格照搬字段数量或章节数量。

### 统一质量门：不判档位，只判是否达标

**不要求模型先判断自己是"轻量"还是"高级"再选择流程。** 所有执行者（不论模型强弱）都执行同一套核心质量门；模型能力强弱只影响门槛之上能不能做得更多，不影响门槛本身——不允许以"我是轻量模型"为由降低门槛，也不允许以"我是高级模型"为由跳过门槛。

**质量地板（所有模型必须满足，硬约束）**：

1. 不丢失用户的明确需求。
2. 不静默把`必须`改成`可选`。
3. 不静默把"自动执行"改成"只给建议"。
4. 不静默把"全部范围"缩小成"代表性样本"。
5. 不虚构模型、价格、参数、现有能力或测试结果。
6. 模糊事实用澄清、占位符、假设或"待探测"表达，不用编造值填充。
7. 每项强制要求都能追溯到提示词里的具体条款（对应需求覆盖账本）。
8. 每项工程要求都有验收方法，或明确标记为"目标"而非"硬性要求"。
9. 涉及账号、费用、外部调用或写操作时，补足授权边界章。
10. 最终产出必须是可直接复制使用的完整提示词。

**能力天花板（高级模型可以自由发挥，不是义务，不需要照搬字段数量）**：

1. 更准确的领域概念分层（如 host/executor、requested/actual 这类原创区分）。
2. 补上用户没说但由上下文必然产生的约束。
3. 更优的 schema、状态机、矩阵或恢复机制。
4. 更强的失败路径建模、更合理的风险分配。
5. 更紧凑或更清晰的组织方式；对需求本身做反例攻击和矛盾检查。
6. 比模板更优的等价结构。

天花板不设上限，但任何扩展仍受以下约束：结构可以创造，事实不能创造；新增内容必须真的改变执行行为、验证方式或风险控制，不是纯装饰；不得覆盖或改变用户已明确的决定；重要推断必须标明依据。

### 授权边界章（按风险启用）

仅当任务涉及账号、凭据、费用、外部调用、写入、删除、发布、推送或其他高修复成本动作时加入本章；纯文本、只读且低风险的规格任务不为凑结构强制加入。需要时套用：

> 本任务授权：读取/修改 <明确对象清单>；使用 <本人已登录/已有权的账号与工具>；在 <隔离环境> 执行 <低风险动作>。
> 本任务不授权：绕过权限/额度/安全策略；读取或输出凭据；<commit/push/发布等按需列出>；静默更换模型/供应商/计费通道。
> 受阻时：如实记录并停止，不隐藏、不改写意图规避判断。

### 模式 D 专属最低回执

除通用「日志与完成回执」模板外，模式 D 交付时必须额外包含：

```markdown
模式 D 回执：
- 主模式 / 辅助模式：<如 B+D>
- 原料状态：<已提供内容 / 已提供路径未读取 / 待补充事实，各项对应哪些需求>
- 需求覆盖摘要：<需求覆盖账本压缩表，每条需求的状态>
- 被扩展的模糊要求：<哪些模糊表述被转成了什么结构>
- 使用的结构类型：<矩阵/状态机/schema/验收清单等，及为什么用这些>
- 事实来源与待验证项：<每个事实项的【原料】/【探测】/【待验证】标注>
- 完整可复制提示词：<全文，代码块包裹>
- 规格密度说明：<为什么是当前长度，删过什么>
- 未执行声明：<仅产出，未派发；或见「交付方式」执行摘要>
- 下一步验证建议：<客户拿到规格后应该先核对什么>
```
